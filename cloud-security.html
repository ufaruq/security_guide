Cloud Security

Comprehensive AWS & GCP security reference for architects, pentesters, and defenders.

1. Shared Responsibility Model

Every cloud engagement begins with a single question: who is responsible for what? Both AWS and GCP publish shared responsibility models that draw a line between what the provider secures and what the customer must secure. That line shifts depending on the service model.

Cloud does not equal secure by default. Providers secure the infrastructure of the cloud. You secure everything you put in the cloud. Misconfigured S3 buckets, overly permissive IAM policies, and unencrypted databases are customer-side failures, not provider failures. The vast majority of cloud breaches stem from customer misconfiguration.

Responsibility by Service Model

Layer IaaS (EC2 / GCE) PaaS (RDS, App Engine) Managed Services (S3, BigQuery)
Physical / Host Provider Provider Provider
Hypervisor Provider Provider Provider
OS Patching Customer Provider Provider
Runtime / Middleware Customer Shared Provider
Application Code Customer Customer N/A (customer owns calling code)
Data Classification Customer Customer Customer
IAM / Access Control Customer Customer Customer
Network Config Customer Shared Provider
Encryption (at rest) Customer Shared (default keys) Shared (default keys)

AWS vs GCP Terminology

Concept AWS GCP

Compliance framework AWS Artifact Google Cloud Compliance Reports
Security posture mgmt Security Hub Security Command Center (SCC)
Trusted advisor AWS Trusted Advisor Active Assist / Recommender
GCP's "Shared Fate" nuance: Google explicitly states they aim for shared fate rather than simply shared responsibility. This means GCP attempts to provide secure-by-default configurations (e.g., uniform bucket-level access, VPC Service Controls) so that misconfiguration is harder. The responsibility boundary is similar in practice, but the defaults are often stricter.

2. Organizational Structure

A well-designed organizational hierarchy is the foundation of cloud security at scale. Both AWS and GCP provide resource hierarchy mechanisms that enable centralized governance, billing isolation, and blast-radius containment. Getting this right from the start prevents years of technical debt.

AWS: Organizations, OUs, and Accounts

AWS Organizations provides a tree structure: a single Management Account (formerly "Master") at the root, with Organizational Units (OUs) that contain member accounts. Each account is a hard security and billing boundary.

Key Components

  • Management Account: The root of the organization. Hosts the billing console and SCPs. Should contain no workloads — only governance tooling.
  • Organizational Units (OUs): Logical groupings. Nest up to 5 levels deep. SCPs applied to an OU inherit downward to all child OUs and accounts.
  • Service Control Policies (SCPs): JSON policies that set the maximum permissions boundary for every principal in an account. They do not grant permissions — they restrict them. SCPs do not affect the management account.
  • AWS Control Tower: An opinionated landing zone service that provisions Organizations, OUs, accounts, guardrails, and SSO automatically. Uses Account Factory for Terraform (AFT) or the built-in Account Factory to vend new accounts.
  • Account Factory: A self-service mechanism (via Service Catalog or AFT) for creating pre-configured accounts that conform to organizational baselines. Teams request accounts; pipelines provision them with standard VPCs, IAM roles, logging, and guardrails.

GCP: Organization, Folders, and Projects

GCP uses a similar tree: an Organization node (tied to a Cloud Identity or Google Workspace domain), containing Folders (nestable up to 10 levels), which contain Projects. Projects are the fundamental resource boundary in GCP — all resources (VMs, buckets, datasets) live inside a project.

Key Components

  • Organization Node: The root, automatically created when a domain is verified. Organization-level IAM and Org Policies propagate downward.
  • Folders: Grouping mechanism for projects. Map to teams, environments, or business units. IAM bindings and Org Policies on a folder inherit downward.
  • Projects: The billing and resource boundary. Each project has a unique ID and number. Quotas, APIs, and IAM are project-scoped.
  • Organization Policies: Constraints applied at the org, folder, or project level. Use boolean or list constraints to restrict allowed values. Examples: restrict VM external IPs, enforce uniform bucket-level access, restrict resource locations.
  • Assured Workloads: Creates compliance-enforced folders that automatically apply Org Policies, data residency, and personnel controls for regulatory frameworks (FedRAMP, IL4, HIPAA).

Guardrails: Preventive vs Detective

Type AWS Mechanism GCP Mechanism Purpose
Preventive SCPs Organization Policies Block forbidden actions before they happen
Preventive Permission Boundaries IAM Deny Policies Cap maximum permissions for principals
Detective AWS Config Rules SCC / Security Health Analytics Detect non-compliant resource configurations
Detective GuardDuty Event Threat Detection Detect active threats and anomalies

Billing Isolation and Blast Radius

In AWS, each account is a hard blast-radius boundary. An IAM compromise in one account cannot directly reach resources in another (unless explicit cross-account trust exists). Billing is consolidated at the org level but traceable per account. In GCP, projects serve a similar function, but IAM inheritance from folders means a folder-level binding grants access to all child projects. This makes folder-level IAM bindings powerful but dangerous.

Account/Project Vending with IaC: Use Terraform (or Pulumi/CDK) to automate account/project creation. AWS Account Factory for Terraform (AFT) and the GCP Project Factory Terraform module both provide standardized pipelines. Every new account/project should be born with: a baseline VPC (or Shared VPC attachment), centralized logging, guardrail policies, and a break-glass role.

3. Identity and Access Management

IAM is the single most critical control plane in any cloud environment. Misconfigured IAM is the root cause of the majority of cloud breaches. Both AWS and GCP use policy-based access control, but their models differ in important ways.

AWS IAM Model

Core Concepts

  • Users: Long-lived identities with access keys or console passwords. Avoid where possible; prefer roles.
  • Groups: Collections of users. Policies attached to groups apply to all members.
  • Roles: Temporary credential identities. EC2 instance profiles, Lambda execution roles, and cross-account access all use roles. Principals assume roles via sts:AssumeRole.
  • Policies: JSON documents with Effect/Action/Resource/Condition. Types: identity-based (attached to user/group/role), resource-based (attached to S3/KMS/etc), permission boundaries, SCPs, session policies.
  • Permission Boundaries: An advanced feature that sets the maximum permissions an IAM entity can have, regardless of identity-based policies. Used for delegation: allow teams to create roles but cap their power.
  • OIDC Federation: GitHub Actions, GitLab CI, and other IdPs can assume AWS roles via OIDC without storing long-lived credentials. The trust policy validates the JWT issuer, audience, and subject claims.

AWS Policy Evaluation Logic

  1. If any explicit Deny exists (in any policy type) → DENY.
  2. If the SCP does not allow the action → DENY (implicit).
  3. If a resource-based policy grants access to the caller directly → ALLOW (for same-account; cross-account requires both sides).
  4. If the permission boundary does not allow it → DENY.
  5. If the session policy does not allow it → DENY.
  6. If an identity-based policy allows it → ALLOW.
  7. Otherwise → DENY (implicit default).

GCP IAM Model

Core Concepts

  • Members (Principals): Google accounts, service accounts, Google groups, Cloud Identity domains, or allUsers / allAuthenticatedUsers.
  • Roles: Collections of permissions. Three types:
    • Basic roles: Owner, Editor, Viewer. Extremely broad — avoid in production.
    • Predefined roles: Fine-grained, maintained by Google (e.g., roles/storage.objectViewer).
    • Custom roles: User-defined permission sets. Useful when predefined roles are too broad.
  • IAM Bindings: A binding ties a member to a role on a resource. Bindings exist at the resource level (project, folder, org, bucket, etc.) and inherit downward through the hierarchy.
  • Service Accounts: Machine identities. Attached to GCE instances, Cloud Functions, GKE pods. Each has a unique email. The iam.serviceAccounts.actAs permission authorizes attaching a service account to a resource (e.g., deploying a VM or Cloud Function with that SA), which indirectly grants that SA's privileges. True impersonation uses generateAccessToken via the Service Account Credentials API.
  • Workload Identity Federation: Allows external identities (AWS, Azure AD, OIDC, SAML) to impersonate GCP service accounts without exporting keys. The gold standard for CI/CD pipelines.
  • IAM Deny Policies: Explicit deny rules that override allow bindings. Similar in effect to AWS SCPs. Applied at org, folder, or project level.

GCP IAM Policy Evaluation Logic

  1. If any IAM Deny Policy matches → DENY.
  2. If the member has a role binding at any level in the hierarchy that grants the permission → ALLOW.
  3. Otherwise → DENY (implicit).
Organization Policies are a separate system. Organization Policies operate on a separate plane from IAM — they constrain resource configurations (e.g., "no external IPs", "only allow resources in us-east1") rather than controlling who can perform actions.
Key difference: AWS evaluates identity policies and resource policies as separate policy types with specific interaction rules. GCP uses a single additive model where bindings at any level in the hierarchy grant access, and you cannot revoke an inherited binding at a lower level — you must use IAM Deny Policies.

Least Privilege Tooling

Tool AWS GCP
Unused permission detection IAM Access Analyzer (unused access) IAM Recommender
Policy generation Access Analyzer policy generation (from CloudTrail) IAM Recommender (role suggestions)
External access detection Access Analyzer (external access) IAM Analyzer (in SCC Premium)
Credential report IAM Credential Report Policy Intelligence
Attack Vectors
  • AWS — iam:PassRole + lambda:CreateFunction: A user who can pass any role to Lambda and create functions can execute code as any role in the account. This is the most common AWS privesc path.
  • AWS — iam:CreatePolicyVersion: A user who can create a new version of any managed policy and set it as default can grant themselves *:*.
  • AWS — Confused Deputy: If a role trust policy uses only aws:SourceAccount without aws:SourceArn, any resource of that service type within your account can trigger the role assumption — not just the intended resource.
  • GCP — iam.serviceAccounts.actAs: A user with actAs permission on a service account and the ability to create compute resources can effectively become that service account.
  • GCP — setIamPolicy: A user with setIamPolicy on a project can grant themselves Owner. This is equivalent to full admin access.
  • GCP — Leaked SA Keys: Exported service account JSON keys found in code repos, CI logs, or config files provide persistent access with no expiry until rotated.

Defences

  • Apply SCPs to deny iam:CreateUser, iam:CreateAccessKey in all accounts except the identity account.
  • Use permission boundaries on all developer-created roles to cap privileges.
  • Require aws:SourceArn in all cross-service role trust policies (confused deputy prevention).
  • Use condition keys: iam:PassedToService, iam:PermissionsBoundary, aws:RequestedRegion.
  • In GCP, use IAM Deny Policies to block setIamPolicy at the org level except for break-glass principals.
  • Never export GCP service account keys. Use Workload Identity Federation for external access.
  • Enforce short-lived credentials everywhere: AssumeRole sessions (max 1hr for sensitive roles), GCP access tokens (1hr default).
  • Run IAM Access Analyzer / IAM Recommender monthly to prune unused permissions.

4. Networking

Cloud networking is software-defined and API-driven. This gives defenders granular control but also exposes a larger attack surface than traditional networking — a single API call can open a port to the internet.

Core Concepts

Concept AWS GCP
Virtual Network VPC (regional) VPC (global, subnets are regional)
Subnets Availability Zone scoped Region scoped
Route Tables Per-subnet route tables VPC-level routes + network tags
NAT NAT Gateway (per-AZ) Cloud NAT (per-region)
Stateful Firewall Security Groups (instance-level) VPC Firewall Rules (network-level)
Stateless Firewall Network ACLs (subnet-level) No direct equivalent (all GCP firewalls are stateful). Hierarchical Firewall Policies provide org/folder-level rules but are stateful.
Private API Access VPC Endpoints (Gateway + Interface) Private Google Access + Private Service Connect
Service Perimeter N/A (use VPC Endpoints + policies) VPC Service Controls
Private Connectivity AWS PrivateLink Private Service Connect
Cross-VPC VPC Peering, Transit Gateway VPC Peering, Shared VPC

AWS: Security Groups vs NACLs

Security Groups are stateful, instance-level firewalls. They allow rules only (no explicit deny). If you allow inbound TCP/443, the response is automatically allowed. Security groups can reference other security groups by ID, enabling service-to-service rules without IP management.

Network ACLs are stateless, subnet-level firewalls. They support both allow and deny rules, evaluated in order by rule number. Both inbound and outbound rules must be explicitly configured. Use NACLs as a coarse defense layer; use Security Groups for fine-grained control.

GCP: Firewall Rules and VPC Service Controls

VPC Firewall Rules are stateful and network-level. They can target instances by network tags or service accounts. Priority-based evaluation (lower number = higher priority). Support both allow and deny rules.

Hierarchical Firewall Policies can be applied at the org or folder level and enforce rules across all child VPCs. This is GCP's equivalent of a centralized firewall — ideal for org-wide blocks.

VPC Service Controls create a security perimeter around GCP API resources. Even if an IAM policy allows access, VPC-SC can block API calls from outside the perimeter. This prevents data exfiltration by limiting which networks, identities, and projects can access sensitive APIs like BigQuery, GCS, and Pub/Sub.

Attack Vectors
  • SSRF to IMDS: An SSRF vulnerability on an EC2 instance can be used to query http://169.254.169.254/latest/meta-data/iam/security-credentials/ and steal the instance role's temporary credentials. The Capital One breach (2019) exploited exactly this path.
  • GCP Metadata Server: Similarly, http://metadata.google.internal/computeMetadata/v1/ exposes access tokens, project info, and service account credentials. The Metadata-Flavor: Google header requirement provides minimal protection.
  • Permissive 0.0.0.0/0 Rules: Security groups or firewall rules allowing all inbound traffic from 0.0.0.0/0 on management ports (22, 3389) are a leading cause of compromise.
  • Unintended Public Exposure: Public subnets with auto-assign public IP, or GCP instances with external IPs, combined with permissive firewall rules.
  • DNS Rebinding: Bypassing metadata endpoint protections by resolving a DNS name to 169.254.169.254.

Defences

  • Enforce IMDSv2 on all EC2 instances: Requires a PUT request with a hop-limited token, blocking SSRF via Layer 7 proxies. Set HttpTokens: required and HttpPutResponseHopLimit: 1.
  • GCP Metadata Concealment: On GKE, enable metadata concealment to prevent pods from accessing the node's metadata server. Use Workload Identity instead.
  • VPC Service Controls: Create perimeters around data services. Even with valid credentials, access from outside the perimeter is denied.
  • Network Segmentation: Separate public, application, and data tiers into distinct subnets. Use NACLs/hierarchical firewalls as a secondary layer.
  • VPC Endpoints / Private Google Access: Route API traffic internally rather than through the public internet.
  • Use SCP / Org Policy to prevent creation of public-facing resources (e.g., deny ec2:RunInstances without IMDSv2, deny external IPs in GCP).

5. Encryption & KMS

Cloud encryption relies on envelope encryption: data is encrypted with a Data Encryption Key (DEK), and the DEK is encrypted with a Key Encryption Key (KEK) managed by a KMS. This approach allows large-scale encryption without sending bulk data to the KMS.

Envelope Encryption Model

  +-------------------+      +--------------------+
  |   Plaintext Data  |      |      KMS (HSM)     |
  +--------+----------+      +--------+-----------+
           |                           |
           v                           v
  +--------+----------+      +--------+-----------+
  | Encrypt with DEK  |      | Generate DEK       |
  | (AES-256-GCM)     |      | (plaintext + enc)  |
  +--------+----------+      +--------+-----------+
           |                           |
           v                           v
  +--------+----------+      +--------+-----------+
  | Ciphertext Data   |      | Encrypted DEK      |
  +--------+----------+      | (wrapped by KEK)   |
           |                  +--------+-----------+
           v                           |
  +--------+-----+----+---------------+
  | Store Together:                    |
  | [ Encrypted DEK | Ciphertext ]    |
  +------------------------------------+

  Decryption:
  1. Send encrypted DEK to KMS
  2. KMS decrypts DEK using KEK (key never leaves HSM)
  3. Use plaintext DEK to decrypt data locally
  4. Discard plaintext DEK from memory

AWS KMS vs GCP Cloud KMS

Feature AWS KMS GCP Cloud KMS
Key Types Symmetric (AES-256), Asymmetric (RSA, ECC), HMAC Symmetric (AES-256), Asymmetric (RSA, EC), MAC
Key Hierarchy AWS-managed, Customer-managed (CMK), Custom Key Store (CloudHSM) Google-managed, CMEK, CSEK, Cloud HSM, Cloud EKM
Access Control Key Policy (resource-based) + IAM policies + Grants IAM bindings on key rings and keys
Automatic Rotation Annual (configurable 90-2560 days) Configurable (any period)
Multi-Region Multi-Region Keys (replicated) Not built-in (use separate keys per region)
External Key Store External Key Store (XKS) Cloud External Key Manager (EKM)
Audit CloudTrail logs every API call Cloud Audit Logs for all key operations

Encryption at Rest Options

Option Description Who Holds Key
Default Encryption Provider encrypts at rest with provider-managed keys. No configuration needed. S3, GCS, EBS, Persistent Disks all do this by default. Provider (fully managed)
CMEK Customer-Managed Encryption Key. You create and manage the key in KMS; the service uses it for encryption. You control rotation, access policy, and can revoke access by disabling the key. Customer (in cloud KMS)
CSEK Customer-Supplied Encryption Key. You provide the raw key material for each API call. The cloud never persistently stores the key. Supported by GCS and GCE disks; AWS equivalent is SSE-C for S3. Customer (on-premises)
When to use CMEK: Whenever you need to enforce separation of duties (the DBA cannot decrypt without the security team enabling the key), meet compliance requirements for key control, or enable crypto-shredding (destroy the key to render all data irrecoverable).
Attack Vectors
  • Permissive Key Policies: An AWS KMS key policy with "Principal": "*" and no condition keys allows any AWS principal to use the key. Combined with a public S3 bucket, this means anyone can decrypt the data.
  • Unencrypted Snapshots: EBS snapshots and GCE disk snapshots can be shared cross-account or made public. If they are not encrypted with a CMEK, the new account owner can access the data freely.
  • Key Policy Privilege Escalation: If a user has kms:PutKeyPolicy on a key, they can modify the key policy to grant themselves decrypt access.
  • Cross-Account Key Grants: AWS KMS Grants can delegate decryption to external accounts. If grants are not audited, persistent access can be hidden.

Defences

  • Apply least privilege to key policies: separate key administrators (manage, rotate) from key users (encrypt, decrypt).
  • Use separate CMEKs per environment (dev/staging/prod) to prevent cross-environment decryption.
  • Enable automatic key rotation (AWS: 365 days default; GCP: set a rotation schedule).
  • Monitor kms:Decrypt and kms:CreateGrant calls in CloudTrail; alert on unusual volume or cross-account calls.
  • Use SCP to deny kms:PutKeyPolicy except for the security tooling account.
  • In GCP, use Org Policy constraints/gcp.restrictCmekCryptoKeyProjects to limit which projects can host CMEK keys.
  • Require CMEK for all data services via preventive policies or Config Rules.

6. Compute Security

Compute workloads span VMs, serverless functions, and containers. Each model shifts the security boundary differently, but the core principles remain: enforce least-privilege identity, harden the runtime, and minimize attack surface.

See also: Kubernetes Security for isolation models, mTLS, and admission policy applied inside clusters; the two pages pair well for containerized workloads.

Virtual Machines: EC2 and GCE

Security Concern AWS EC2 GCP GCE
Instance Identity Instance Profile (IAM Role) Service Account (attached to VM)
Metadata Protection IMDSv2 (token-based, hop-limited) Metadata-Flavor header requirement
Patching Systems Manager Patch Manager OS Patch Management (VM Manager)
Integrity Monitoring Nitro TPM / Trusted Boot Shielded VMs (vTPM, Secure Boot, Integrity Monitoring)
Confidential Computing Nitro Enclaves, AMD SEV-SNP on EC2 Confidential VMs (AMD SEV, SEV-SNP, Intel TDX)
SSH Access EC2 Instance Connect, SSM Session Manager OS Login, IAP TCP Forwarding

Serverless: Lambda and Cloud Functions

Serverless shifts OS management to the provider, but application-level risks remain. The execution role/service account determines what cloud APIs the function can call. Over-privileged serverless functions are a major risk because they are often event-triggered with user-controlled input.

  • Execution Role Scope: Each Lambda function should have its own dedicated IAM role with only the permissions it needs. Reusing roles across functions violates least privilege.
  • Event Injection: Functions triggered by API Gateway, S3, SQS, or Pub/Sub receive event payloads that may contain user-controlled data. If this data is passed to shell commands, SQL queries, or downstream APIs without validation, injection attacks are possible.
  • Secrets Management: Never embed secrets in function code or environment variables in plaintext. Use AWS Secrets Manager, GCP Secret Manager, or SSM Parameter Store (SecureString).
  • Cold Start and Execution Environment: Lambda reuses execution environments across invocations. Data left in /tmp or global variables persists and may leak between tenants in multi-tenant designs.

Containers: EKS and GKE

Managed Kubernetes adds a control-plane layer managed by the provider, but the workload pods, RBAC, and network policies are the customer's responsibility.

  • Pod Security: Use Pod Security Standards (Restricted profile) to enforce non-root containers, read-only root filesystems, and dropped capabilities. Replace the deprecated PodSecurityPolicy with Pod Security Admission or a policy engine like Kyverno/Gatekeeper.
  • Workload Identity: Both EKS and GKE support mapping Kubernetes service accounts to cloud IAM roles/service accounts. This eliminates the need for node-level credentials that all pods on a node would share.
    • EKS: IAM Roles for Service Accounts (IRSA) via OIDC federation, or EKS Pod Identity.
    • GKE: Workload Identity maps a KSA to a GCP service account.
  • Image Scanning: Scan images in ECR (via Amazon Inspector) or Artifact Registry (via Artifact Analysis / Container Scanning) before deployment.
  • Binary Authorization (GKE): Enforces that only signed, attested images can be deployed. Uses attestations stored in Artifact Analysis to verify provenance.
  • Network Policies: Kubernetes NetworkPolicy objects restrict pod-to-pod traffic. Both EKS (with Calico or VPC CNI network policy) and GKE (with Dataplane V2 / Calico) support them. Default-deny and allow only required paths.
Attack Vectors
  • IMDS Credential Theft: An SSRF or RCE on an EC2/GCE instance allows querying the metadata endpoint for temporary credentials. On EC2 with IMDSv1, a simple GET request suffices. On GCE, the Metadata-Flavor: Google header is trivially added.
  • Over-Privileged Lambda: A Lambda with s3:* and iam:* permissions that processes user-uploaded files becomes a full-privilege gateway if the handler has an injection flaw.
  • Container Escape: Running containers as root with --privileged or with SYS_ADMIN capability allows kernel exploits to escape to the host node, accessing kubelet credentials and other pods.
  • Node-Level SA Theft (GKE): Without Workload Identity, every pod on a GKE node can access the node's GCE service account token via the metadata server. One compromised pod compromises all.
  • Kubelet API: If the kubelet API (port 10250) is exposed without authentication, attackers can exec into any pod on that node.

Defences

  • Enforce IMDSv2 with HttpPutResponseHopLimit: 1 on all EC2 instances. Use an SCP to deny ec2:RunInstances unless ec2:MetadataHttpTokens is required.
  • In GKE, enable Workload Identity and metadata concealment on all clusters. Remove the default compute service account from node pools.
  • Give every Lambda / Cloud Function a dedicated, minimally-scoped execution role.
  • Run all containers as non-root with a read-only root filesystem and dropped ALL capabilities.
  • Enforce image signing: Binary Authorization (GKE) or a Kyverno policy that verifies cosign signatures (EKS).
  • Use Kubernetes NetworkPolicy to default-deny all ingress and egress, then allow specific paths.
  • Scan images in CI/CD and block deployment if critical CVEs are found.

7. Storage Security

Object storage (S3 and GCS) is the most commonly misconfigured cloud service. Public bucket exposures have caused some of the most damaging cloud breaches. Both providers have added multiple layers of protection — but only if you enable them.

Amazon S3 Security

Access Control Layers

Note: These layers are evaluated collectively, not sequentially. An explicit deny in any layer overrides allows elsewhere. Block Public Access acts as a hard override on top of all other policies.
  • S3 Block Public Access: An account-level or bucket-level setting that overrides any policy or ACL that would grant public access. This is the most important single control.
  • Bucket Policy: Resource-based JSON policy attached to the bucket. Controls who can access what objects and under what conditions.
  • IAM Policies: Identity-based policies on users/roles. Combined with bucket policies using the standard policy evaluation logic.
  • ACLs (Legacy): Object-level and bucket-level ACLs. AWS recommends disabling ACLs entirely (BucketOwnerEnforced) and using policies instead.
  • Access Points: Named network endpoints with their own policy, scoped to a VPC or internet. Simplify access management for shared buckets.

Pre-Signed URLs

Pre-signed URLs grant time-limited access to a specific S3 object. The URL inherits the permissions of the IAM principal who generated it. Set short expiration times (minutes, not hours). The URL becomes invalid if the signing principal's permissions are revoked.

Google Cloud Storage Security

Access Control

  • Uniform Bucket-Level Access: Disables ACLs entirely and uses only IAM for access control. Google recommends this for all buckets. Once enabled for 90 days, it becomes permanent.
  • IAM Bindings: Grant roles on buckets or projects. roles/storage.objectViewer, roles/storage.objectCreator, etc.
  • Signed URLs: Equivalent to S3 pre-signed URLs. Created by a service account; time-limited.
  • allUsers / allAuthenticatedUsers: Special IAM members that make buckets/objects public. Org Policies can block these.
Attack Vectors
  • Public Buckets: A bucket policy with "Principal": "*" and no condition keys makes the bucket world-readable. Attackers use tools like grayhatwarfare.com and bucket enumeration scripts to discover these.
  • ACL Misconfiguration: Legacy ACLs granting s3:GetObject to AllUsers or AuthenticatedUsers. The "AuthenticatedUsers" group includes any AWS account, not just your org.
  • Overly Broad Principal: A bucket policy that uses "Principal": {"AWS": "*"} without a condition key allows any authenticated AWS principal to access the bucket.
  • Write Access Abuse: If an attacker can write to an S3 bucket that hosts static content, CloudFormation templates, or Lambda deployment packages, they can inject malicious payloads.
  • Bucket Name Squatting: If you delete a bucket and an attacker recreates it with the same name, any client still referencing that bucket name sends data to the attacker.
Case Study: Capital One (2019). An SSRF vulnerability in a WAF/reverse proxy running on an EC2 instance allowed the attacker to query the IMDSv1 metadata service. The attacker queried the EC2 metadata service (IMDSv1) to obtain temporary credentials for an IAM role. That role had overly broad S3 permissions, allowing the attacker to exfiltrate 100+ million customer records from S3 buckets. The root causes: IMDSv1 (no SSRF protection), an over-privileged IAM role, and insufficient network segmentation. This breach drove AWS to aggressively promote IMDSv2 and S3 Block Public Access.

Defences

  • Enable S3 Block Public Access at the organization level via SCP. This single control prevents the most common S3 exposure vector.
  • Disable ACLs on all S3 buckets (set ObjectOwnership: BucketOwnerEnforced).
  • Enable Uniform Bucket-Level Access on all GCS buckets. Use an Org Policy to enforce this.
  • Use GCP Org Policy constraints/storage.publicAccessPrevention set to enforced at the org level.
  • Enable S3 server access logging or CloudTrail data events for sensitive buckets.
  • Enable GCS data access audit logs for sensitive buckets.
  • Use S3 Object Lock (WORM) for compliance data that must be immutable.
  • Require aws:SecureTransport condition in bucket policies to deny non-HTTPS requests.
  • Use VPC endpoints / Private Google Access so storage traffic never traverses the public internet.

8. Logging & Detection

Visibility is the foundation of cloud defense. You cannot detect what you do not log, and you cannot investigate what you cannot query. Both AWS and GCP provide comprehensive logging services, but they must be centralized, protected, and monitored.

AWS Logging Stack

Service What It Captures Key Notes
CloudTrail API calls (management events + optional data events) Organization trail covers all accounts. Data events for S3/Lambda cost extra but are critical for forensics.
VPC Flow Logs Network metadata (src/dst IP, port, protocol, bytes, accept/reject) Enable on all VPCs and subnets. Does not capture packet contents. Enriched format includes VPC ID, subnet ID, instance ID.
GuardDuty Threat detection (DNS, CloudTrail, VPC flows, S3, EKS, Lambda, RDS) ML-based anomaly detection. Enable in all regions and all accounts via delegated admin. Findings feed into Security Hub.

AWS vs GCP Logging Comparison

Feature AWS GCP
API Audit Trail CloudTrail (org trail) Cloud Audit Logs (org sink)
Threat Detection GuardDuty SCC Premium (Event Threat Detection)
Posture Management Security Hub + Config SCC (Security Health Analytics)
SIEM Amazon Security Lake + OpenSearch / 3rd party Chronicle

Config Drift Detection AWS Config SCC + Cloud Asset Inventory
Automated Remediation Config Remediation, EventBridge + Lambda SCC Notifications + Cloud Functions
Attack Vectors
  • CloudTrail Tampering: An attacker with cloudtrail:StopLogging or cloudtrail:DeleteTrail can blind defenders. This is the first action sophisticated attackers take. Even updating a trail to exclude specific event types can evade detection.
  • Log Evasion: Some AWS actions are not logged by CloudTrail by default (e.g., certain S3 data events, CloudWatch Logs:GetLogEvents). Attackers can operate in these blind spots.
  • GuardDuty Suppression: An attacker with guardduty:CreateFilter can create a filter that auto-archives specific finding types, effectively suppressing alerts.
  • GCP Audit Log Gaps: Data Access logs are not enabled by default. Without them, an attacker who reads data (BigQuery queries, GCS object reads) leaves no trace.
  • Log Deletion: If CloudTrail logs are stored in an S3 bucket without Object Lock or MFA Delete, an attacker with S3 permissions can delete the evidence.

Defences

  • Immutable Logging: Store CloudTrail logs in a dedicated log archive account. Enable S3 Object Lock (Governance or Compliance mode) on the log bucket. Use MFA Delete. The log archive account should have an SCP that denies all actions except read-only access by the SIEM.
  • Organization-Level Trails / Sinks: Create a CloudTrail Organization Trail or GCP org-level log sink. Individual accounts/projects cannot disable these.
  • Enable Data Events / Data Access Logs: For sensitive S3 buckets and GCS buckets, enable data event logging. In GCP, enable Data Access audit logs for BigQuery, GCS, Pub/Sub, and other data services.
  • SIEM Integration: Forward all logs to a SIEM (Splunk, Elastic, Chronicle, Amazon Security Lake). Build detection rules for the alert table above.
  • Automated Remediation: Use EventBridge + Lambda (AWS) or SCC Notifications + Cloud Functions (GCP) to automatically remediate common issues (e.g., close public security groups, revoke public bucket access).
  • SCP / Org Policy: Deny cloudtrail:StopLogging, cloudtrail:DeleteTrail, guardduty:DeleteDetector, guardduty:CreateFilter except from the security account.
