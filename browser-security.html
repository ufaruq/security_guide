Browser Security Model

Same-Origin Policy, CORS, CSP, cookies, isolation and side-channel mitigations

The browser is a complex security boundary. It runs code from dozens of origins simultaneously, enforces access controls between them, and must protect sensitive data even when hosting malicious third-party scripts. Understanding the browser security model means understanding how these boundaries work, where they break, and how to configure them correctly.

Same-Origin Policy

The Same-Origin Policy (SOP) is the browser's fundamental security boundary. Two URLs share an origin if and only if they have the same scheme, host, and port. SOP restricts how documents and scripts loaded from one origin can interact with resources from another origin.

Origin Comparison

https://example.comhttp://example.comNoDifferent scheme
https://example.comhttps://sub.example.comNoDifferent host
https://example.com:443https://example.com:8443NoDifferent port
https://example.comhttps://example.com:443Yes443 is implicit for HTTPS

What SOP Allows and Blocks

Allowed cross-origin:
  - Link navigation (<a href>)
  - Form submission (POST — though CSRF risk)
  - Embedding (<img>, <script>, <iframe>, <video>, <link rel="stylesheet">)
  - window.open() / window.location navigation

Blocked cross-origin (without CORS):
  - Reading response body from fetch/XHR
  - Reading iframe's contentDocument or contentWindow properties
  - Reading cookies, localStorage, sessionStorage
  - Calling methods on another origin's window (except postMessage)
SOP blocks reads, not writes. An attacker page can send a POST request to your banking site (CSRF) — SOP won't stop that. SOP only prevents the attacker from reading the response.

CORS (Cross-Origin Resource Sharing)

CORS is a controlled relaxation of SOP. Servers opt-in to allowing specific cross-origin reads by including Access-Control-Allow-* response headers. The browser enforces these headers — they are not accessible to client-side code.

See also: CSRF to understand how cross-site requests abuse state, and Kubernetes network policies to keep internal services unreachable even if CORS or auth is misconfigured.

Simple vs Preflighted Requests

Simple request criteria (no preflight):
  - Method: GET, HEAD, POST
  - Content-Type: application/x-www-form-urlencoded | multipart/form-data | text/plain
  - No custom headers

Preflighted request: browser sends OPTIONS first
  - Methods: PUT, DELETE, PATCH, or custom
  - Content-Type: application/json (or other non-simple)
  - Any custom header (e.g., Authorization, X-API-Key)

OPTIONS preflight exchange:
  Request:
    OPTIONS /api/data HTTP/1.1
    Origin: https://app.example.com
    Access-Control-Request-Method: DELETE
    Access-Control-Request-Headers: Authorization

  Response (server grants permission):
    Access-Control-Allow-Origin: https://app.example.com
    Access-Control-Allow-Methods: GET, POST, DELETE
    Access-Control-Allow-Headers: Authorization
    Access-Control-Max-Age: 86400

Credentialed Requests

Cookies sent cross-origin require explicit opt-in on both client and server.

// Client must explicitly request credentials
fetch('https://api.example.com/data', {
  credentials: 'include'  // sends cookies
});

// Server MUST respond with explicit origin (not wildcard) + allow-credentials
// Access-Control-Allow-Origin: https://app.example.com   ← exact origin
// Access-Control-Allow-Credentials: true
// Access-Control-Allow-Origin: *  ← INVALID for credentialed requests

Common CORS Misconfigurations

Attack Examples
// 1. Origin reflection: server echoes back whatever Origin header is sent
// Vulnerable server code:
app.use((req, res, next) => {
  res.header('Access-Control-Allow-Origin', req.headers.origin);  // DANGEROUS
  res.header('Access-Control-Allow-Credentials', 'true');
  next();
});
// Attacker from any origin gets full access including cookies

// 2. Wildcard + credentials (browser actually blocks this, but config intent is wrong)
// Access-Control-Allow-Origin: *
// Access-Control-Allow-Credentials: true  ← browser rejects this combination

// 3. Trusting null origin (sandbox iframes send Origin: null)
// Access-Control-Allow-Origin: null  ← dangerous
// Exploited with: <iframe sandbox="allow-scripts" srcdoc="...fetch with credentials...">

// 4. Overly broad origin allowlist with substring check
// Vulnerable: if (origin.includes('example.com')) allow
// Bypassed with: https://evil-example.com or https://example.com.attacker.com

// 5. Trusting all subdomains: /^https:\/\/.*\.example\.com$/
// Bypassed if any subdomain is XSS-vulnerable

Correct CORS Configuration

const ALLOWED_ORIGINS = new Set([
  'https://app.example.com',
  'https://admin.example.com'
]);

app.use((req, res, next) => {
  const origin = req.headers.origin;
  if (ALLOWED_ORIGINS.has(origin)) {
    res.header('Access-Control-Allow-Origin', origin);
    res.header('Vary', 'Origin');  // Important for caching
    res.header('Access-Control-Allow-Credentials', 'true');
  }
  next();
});

Content Security Policy (CSP)

CSP is an HTTP response header that tells the browser which sources it may load resources from. It's the most powerful XSS mitigation available — even if an attacker injects a <script> tag, CSP prevents it from executing.

style-srcCSS'self' 'unsafe-inline'
img-srcImages'self' data: https:

font-srcWeb fonts'self' https://fonts.gstatic.com
frame-src<iframe> sources'none'
frame-ancestorsWho can embed this page'none' or 'self'
object-src<object>, <embed>, and plugin content (Flash reached end-of-life December 31, 2020 and is no longer supported by any major browser; the directive remains important for blocking other plugin types, e.g., PDFs rendered via <object>)'none'

Nonce-based CSP (Recommended)

<!-- Server generates a fresh cryptographic nonce per request -->
<!-- Response header: Content-Security-Policy: script-src 'nonce-r4nd0mB4se64==' -->

<!-- Only scripts with the matching nonce attribute execute -->
<script nonce="r4nd0mB4se64==">
  // This runs
  console.log('Allowed script');
</script>

<!-- Injected scripts (XSS) have no nonce and are blocked -->
<script>alert(1)</script>  <!-- CSP blocks this -->

Allows scripts loaded by a trusted nonce/hash script to load further scripts.
Useful for webpack bundles, SPAs.

Common CSP Bypasses

Bypass Patterns
1. JSONP endpoints on allowlisted domains
   Bypass: <script src="https://allowed.com/jsonp?callback=alert(1)">
   Fix: Remove JSONP endpoints; use Fetch + CORS instead

2. open redirect on allowlisted domain
   Bypass: <script src="https://allowed.com/redirect?url=//attacker.com/evil.js">
   Fix: Don't allowlist domains you don't fully control

3. Angular/template injection (if ng-app is present with 'unsafe-eval')
   Fix: Never use 'unsafe-eval'; disable client-side template compilation in prod

4. Missing base-uri: attacker injects <base href="https://attacker.com/">
   Fix: Always include base-uri 'self' or base-uri 'none'

5. 'unsafe-inline' negates script-src protection
   Fix: Use nonces or hashes instead

6. data: URI in script-src
   Bypass: <script src="data:,alert(1)">
   Fix: Never allow data: in script-src

Report-Only Mode

Content-Security-Policy-Report-Only: default-src 'self'; report-to csp-violations
Reporting-Endpoints: csp-violations="https://csp-report.example.com/csp"

Subresource Integrity (SRI)

When loading third-party scripts or stylesheets from a CDN, SRI verifies the fetched file matches a cryptographic hash you specify. If the CDN is compromised and serves a different file, the browser refuses to execute it.

<!-- SRI attribute: base64-encoded SHA-384 hash of the expected file content -->
<script
  src="https://cdn.example.com/library/v1.2.3/lib.min.js"
  integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K/ux6WldDs4LUbv7EFsXXB3lBtSMXeGv"
  crossorigin="anonymous">
</script>

<!-- Generate hash:
  openssl dgst -sha384 -binary lib.min.js | openssl base64 -A
  or: shasum -a 384 lib.min.js | awk '{print $1}' | xxd -r -p | base64
  or: use https://www.srihash.org/
-->

<!-- crossorigin="anonymous": required to enforce SRI for cross-origin resources
     Without it, browser skips SRI for cross-origin resources -->

<!-- Also works for stylesheets -->
<link rel="stylesheet"
  href="https://cdn.example.com/styles.css"
  integrity="sha384-..."
  crossorigin="anonymous">

Limitations

  • SRI only works for static files — it breaks on dynamically generated responses
  • The integrity attribute is not supported on importScripts() within Service Workers (though the Fetch API can still perform programmatic integrity checks)
  • Versioned CDN URLs are critical — update hash when updating version

Trusted Types

Trusted Types is a browser API that can significantly reduce the risk of DOM XSS by restricting which values can be assigned to dangerous DOM sinks. When Trusted Types is enforced via CSP, any code that tries to set innerHTML, eval, document.write, or similar sinks with a plain string will throw an exception.

Browser support caveat: As of 2026, Trusted Types is supported in Chromium-based browsers (Chrome, Edge) only. Firefox and Safari do not yet support it. In unsupported browsers, the require-trusted-types-for CSP directive is silently ignored, so Trusted Types alone cannot be relied upon as a universal defence — it should be layered with other mitigations (e.g., a sanitisation library) for full coverage. 
// Enforcement via CSP:
// Content-Security-Policy: require-trusted-types-for 'script'; trusted-types default

// Without Trusted Types — any string can go to a dangerous sink:
element.innerHTML = userInput;  // XSS possible

// With Trusted Types enforcement:
element.innerHTML = userInput;  // Throws TypeError: TrustedHTML required

// Create a Trusted Types policy that sanitises before assignment:
const policy = trustedTypes.createPolicy('default', {
  createHTML: (input) => DOMPurify.sanitize(input),  // sanitise first
  createScriptURL: (url) => {
    if (new URL(url).hostname === 'cdn.example.com') return url;
    throw new Error('Untrusted script URL');
  }
});

// Now use the policy — produces a TrustedHTML object, not a plain string:
element.innerHTML = policy.createHTML(userInput);  // Safe

// Dangerous sinks that Trusted Types covers:
// innerHTML, outerHTML, document.write(), document.writeln()
// eval(), Function(), setTimeout(string), setInterval(string)
// script.src, script.text, iframe.srcdoc
// location.href, location.assign() with javascript: URLs

Clickjacking

Clickjacking embeds the target site in a transparent <iframe> overlaid on a decoy page. Victims click buttons on the transparent target while believing they click the decoy. Effective against any page with consequential button clicks.

Attack Structure
<!-- Attacker page structure -->
<style>
  #victim-frame {
    position: fixed;
    top: 0; left: 0;
    width: 100%; height: 100%;
    opacity: 0.0001;   /* invisible but clickable */
    z-index: 2;
    /* position adjusted so the dangerous button aligns with decoy button */
    transform: translateY(-300px);
  }
  #decoy-button {
    position: fixed;
    top: 300px; left: 200px;
    z-index: 1;
  }
</style>
<iframe id="victim-frame" src="https://victim.com/account/settings"></iframe>
<button id="decoy-button">Click to claim your prize!</button>

Clickjacking: Layered Attack Visualization

Bottom Layer: Attacker's Page
Win a Prize!
Top Layer: Target Site (opacity: 0.0001)
Transfer $1000
How it works:
Victim sees "Win a Prize!"
Actually clicks "Transfer $1000"
Target iframe is near-invisible
The attacker positions the transparent target site iframe so that its real button aligns perfectly with the fake decoy button underneath.

Defence

# X-Frame-Options (older, widely supported)
X-Frame-Options: DENY          # no framing allowed anywhere
X-Frame-Options: SAMEORIGIN    # only same origin can frame

# CSP frame-ancestors (supersedes X-Frame-Options, supports multiple origins)
Content-Security-Policy: frame-ancestors 'none';
Content-Security-Policy: frame-ancestors 'self' https://trusted-parent.example.com;

# Both headers are commonly set for broad compatibility

Frame-busting JavaScript (Unreliable)

// This is bypassable — attacker can sandbox the iframe to disable JS
if (window.top !== window.self) {
  window.top.location = window.self.location;  // Breakout attempt
}
// Bypass: <iframe sandbox="allow-scripts allow-same-origin" src="victim">
// The sandbox attribute disables navigation from inner frame
// Use X-Frame-Options / CSP instead — headers cannot be bypassed
JavaScript frame-busting is unreliable. Attackers can use sandbox="allow-scripts" on the embedding iframe to prevent the inner page from navigating the top-level context. Always use HTTP headers.

Cookies Deep Dive

Cookies are the primary session mechanism on the web. Their security attributes control when they're sent, whether JS can read them, and how they behave in cross-site contexts. Misconfigured cookies are a leading cause of session hijacking and CSRF.

Cookie Security Attributes

AttributeEffectWhen to use
SecureCookie only sent over HTTPSAlways for session cookies in production
HttpOnlyJavaScript cannot read this cookie via document.cookieAlways — prevents XSS exfiltration
SameSite=StrictCookie not sent on any cross-site requestMaximum CSRF protection; breaks some OAuth flows
SameSite=LaxCookie sent on top-level GET navigations; not on cross-site POST/fetchGood default since Chrome 80; allows SSO redirects
SameSite=None; SecureCookie sent cross-site; MUST have Secure attributeThird-party cookies, embedding in iframes
Domain=.example.comCookie shared across all subdomainsOnly when subdomains need shared session
Path=/apiCookie only sent for requests to /api prefixIsolate cookies to specific paths
Max-Age=3600Cookie expires after N secondsSession persistence
Expires=...Absolute expiry datePersistent cookies

Cookie Prefixes

# __Host- prefix enforces:
#   - Secure attribute required
#   - Path=/ required (cannot be Path=/api)
#   - No Domain attribute (only exact host, not subdomains)
Set-Cookie: __Host-session=abc123; Secure; Path=/; HttpOnly; SameSite=Lax

# __Secure- prefix enforces:
#   - Secure attribute required
# (weaker than __Host-)
Set-Cookie: __Secure-token=xyz; Secure; Path=/; HttpOnly

# Why this matters: cookie tossing attack
# Subdomain (sub.example.com) sets a plain session cookie for Domain=.example.com
# This overwrites the main site's session cookie — __Host- prefix prevents this

Cookie Jar Overflow / Tossing

Browsers have per-domain cookie limits (~180 cookies in Chrome).
Attacker fills up the jar with junk cookies → legitimate session cookie evicted.
Or: attacker sets a shadowing cookie from a subdomain to override main site cookie.

Mitigations:
  - __Host- prefix prevents cross-subdomain overwriting
  - Cookie partitioning (CHIPS) isolates third-party cookies per top-level site

CHIPS (Cookies Having Independent Partitioned State)

# Third-party cookie that is partitioned per top-level site
Set-Cookie: embed-session=abc; Secure; Path=/; SameSite=None; Partitioned

# The same cookie set on embed.widget.com will have DIFFERENT values
# depending on whether the top-level site is site-a.com or site-b.com
# Prevents cross-site tracking via shared cookie values

Fetch Metadata Headers

Fetch Metadata request headers are automatically added by the browser, telling the server where a request originated and how it was initiated. Servers can use these headers to implement Resource Isolation Policies — rejecting requests that should not be cross-origin.

Header Reference

# Sec-Fetch-Site: relationship between the request origin and the resource origin
Sec-Fetch-Site: same-origin   # same origin
Sec-Fetch-Site: same-site     # same site, different origin (sub.example.com → example.com)
Sec-Fetch-Site: cross-site    # different site
Sec-Fetch-Site: none          # user-initiated navigation (typed URL, bookmark)

# Sec-Fetch-Mode: how the request was initiated
Sec-Fetch-Mode: navigate      # top-level navigation
Sec-Fetch-Mode: cors          # CORS fetch
Sec-Fetch-Mode: no-cors       # cross-origin fetch without CORS (e.g., <img>)
Sec-Fetch-Mode: same-origin   # same-origin fetch
Sec-Fetch-Mode: websocket     # WebSocket upgrade

# Sec-Fetch-Dest: the destination/type of the requested resource
Sec-Fetch-Dest: document      # top-level page navigation
Sec-Fetch-Dest: script        # <script> tag
Sec-Fetch-Dest: image         # <img> tag
Sec-Fetch-Dest: empty         # fetch() API calls and XHR (no specific destination)

# Sec-Fetch-User: true if request was triggered by user activation
Sec-Fetch-User: ?1

Server-side Resource Isolation Policy

def resource_isolation_policy(request):
    """Reject cross-site requests that shouldn't be cross-origin."""
    site = request.headers.get('Sec-Fetch-Site', '')
    mode = request.headers.get('Sec-Fetch-Mode', '')
    dest = request.headers.get('Sec-Fetch-Dest', '')

    # Allow same-origin and browser-direct requests
    if site in ('same-origin', 'same-site', 'none'):
        return True

    # Allow simple cross-site navigations (user clicked a link)
    if mode == 'navigate' and dest == 'document':
        return True

    # Block everything else (CSRF, cross-site XHR, etc.)
    return False  # Return 403

Permissions Policy

The Permissions Policy header (formerly Feature-Policy) controls which browser features and APIs a page and its iframes can use. Disabling unused features limits the damage from XSS — a script cannot exfiltrate microphone recordings if the mic API is disabled.

# Disable camera, mic, geolocation, payment, USB for all origins (including iframes)
Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=(), usb=()

# Allow geolocation only for self, and specific trusted embed
Permissions-Policy: geolocation=(self "https://maps.example.com")

# Available features (non-exhaustive):
# camera, microphone, geolocation, payment, usb, bluetooth
# accelerometer, gyroscope, magnetometer
# autoplay, fullscreen, picture-in-picture
# document-domain, sync-xhr, encrypted-media
# (cross-origin isolation is controlled by COOP/COEP headers, not Permissions Policy)

iframe Delegation

<!-- Iframe inherits parent's Permissions Policy restrictions -->
<!-- To grant a feature to an iframe, use the allow attribute -->
<iframe src="https://maps.example.com/embed"
  allow="geolocation">
</iframe>
<!-- In Permissions Policy, listing a feature in allow grants it to the iframe's origin.
     Omitting a feature denies it (e.g., camera is denied here because it is not listed). -->

Cross-Origin Isolation (COOP/COEP)

Cross-Origin Isolation is a hardened browsing context that enables powerful APIs like SharedArrayBuffer and high-resolution timers — while also preventing Spectre-class side-channel attacks. It requires both COOP and COEP headers.

# Cross-Origin-Opener-Policy: isolates browsing context group
# same-origin: page gets its own browsing context group — cannot be opened by
# or communicate with cross-origin pages via window.opener
Cross-Origin-Opener-Policy: same-origin

# Cross-Origin-Embedder-Policy: requires all subresources to opt-in
# require-corp: all cross-origin resources must send CORP header
#   OR be loaded with crossorigin attribute + CORS approval
Cross-Origin-Embedder-Policy: require-corp

Why COEP is Hard to Deploy

Your page loads:
  - Fonts from Google Fonts → must add Cross-Origin-Resource-Policy: cross-origin
  - Images from S3 → must add CORP header or use crossorigin attribute
  - Iframes from third parties → must add COEP: unsafe-none on the iframe's response

Any cross-origin resource that doesn't opt-in will be blocked.
Use "credentialless" mode to ease adoption:
  Cross-Origin-Embedder-Policy: credentialless
  # Loads cross-origin resources without cookies — less breakage

What Cross-Origin Isolation Unlocks

// SharedArrayBuffer — requires crossOriginIsolated = true
const sab = new SharedArrayBuffer(1024);  // Throws without isolation

// High-resolution performance.now() (required for Spectre exploit)
// Without isolation: resolution reduced to 1ms to prevent timing attacks
// With isolation: full resolution restored (deemed safe due to process isolation)
performance.now();  // μs precision

// Check isolation status:
console.log(crossOriginIsolated);  // true or false

iframe Sandboxing

The sandbox attribute on <iframe> restricts what the embedded content can do. By default, a sandboxed iframe can do almost nothing — capabilities are added back with explicit tokens.

TokenGrants permission to
allow-scriptsExecute JavaScript
allow-same-originTreat content as same origin (needed for cookies/storage)
allow-formsSubmit forms
allow-popupswindow.open(), <a target="_blank">

<iframe src="untrusted-content.html" sandbox></iframe>

<!-- Common configuration for user-generated content preview -->

Danger: sandbox="allow-scripts allow-same-origin" together is extremely dangerous. If the iframe content is same-origin with the embedding page, the script can access parent.document and remove the sandbox attribute itself via DOM manipulation. For cross-origin content, the combination still grants the iframe full origin privileges (cookies, storage), defeating the sandbox's purpose even though direct attribute removal is not possible.

Spectre & Side-Channel Mitigations

The Spectre (CVE-2017-5753/5715) and Meltdown (CVE-2017-5754) vulnerabilities demonstrated that speculative execution in modern CPUs can leak data across security boundaries via timing side channels. In the browser, JavaScript's access to high-resolution timers made it possible to read memory from other browser tabs or the kernel.

The Attack Mechanism (Simplified)

Spectre exploits CPU branch prediction and out-of-order execution:

1. CPU speculatively executes code past a conditional branch before knowing the branch outcome
2. Speculatively executed code accesses memory it shouldn't (e.g., another process's memory)
3. Access loads data into CPU cache
4. Branch resolves as not-taken — CPU rolls back the computation, BUT the cache is not rolled back
5. Attacker measures timing of memory accesses to determine what was loaded into cache
6. Infers the value of memory that should be inaccessible

In the browser:
  - JavaScript uses a SharedArrayBuffer with a worker-thread busy-loop counter to build a high-resolution timer
  - The SharedArrayBuffer counter technique can achieve effective sub-microsecond resolution. Combined with statistical amplification over many measurements, this is sufficient to distinguish cache hits (~1-4ns) from misses (~60-200ns).
  - Read arbitrary memory from the same process (e.g., cross-origin iframe data)

Browser Mitigations

1. Reduced timer precision
   performance.now() → reduced resolution (1ms in Firefox; 100μs in Chrome non-isolated contexts)
   Date.now()        → 1ms resolution
   Resolution is restored to higher precision in cross-origin isolated contexts (crossOriginIsolated = true)
   Impact: makes timing attacks significantly harder

2. Disabled SharedArrayBuffer (Jan 2018); re-enabled only in cross-origin isolated contexts
   (Firefox 79, July 2020; Chrome 92, July 2021)
   SharedArrayBuffer allows building precise timers via a worker-thread busy-loop counter

3. Site Isolation (Chrome, Firefox)
   Each top-level site runs in its own OS process
   Cross-origin iframes → separate process
   Memory pages from different sites cannot share the same process memory

4. Spectre-resistant fetch mode
   'no-cors' requests for cross-origin resources are opaque — response body inaccessible
   Prevents Spectre from being used to read cross-origin response data

5. Cross-Origin Read Blocking (CORB)
   Blocks cross-origin HTML/JSON/XML responses from entering renderer process
   Even if Spectre reads process memory, sensitive data was never loaded

6. Cross-Origin Opener Policy (COOP)
   Prevents window.opener / postMessage channel between cross-origin pages
   Cross-origin pages cannot share a browsing context group → separate processes

What Developers Should Do

Developer Checklist

  • Deploy COOP + COEP for pages that need SharedArrayBuffer (high-perf apps)
  • Ensure sensitive data responses include appropriate CORP headers
  • Don't embed secrets in page source that can be read by any same-process script
  • Use HttpOnly cookies for session tokens (inaccessible to JS even if Spectre reads DOM)
