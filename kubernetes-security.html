Kubernetes Security

Threat model, workload hardening, RBAC, network policy, workload identity, and managed Kubernetes differences

Kubernetes security is less about one magic control and more about reducing blast radius in a fast-moving distributed system. Pods talk to an API server, kubelets run on every node, controllers continuously reconcile state, and cloud identities often sit just one metadata call away. If one workload gets popped, the question is whether the attacker stays trapped inside one namespace or turns that foothold into cluster-wide or cloud-wide access.

This guide treats Kubernetes from the perspective of a defender and a pentester at the same time: what the important trust boundaries are, how attackers chain misconfigurations together, and what a modern hardening baseline looks like on upstream Kubernetes plus managed platforms like GKE, EKS, and AKS.

Example style note: Infrastructure examples on this page use Terraform where Terraform is the natural provisioning layer. Short kubectl examples that remain are operational checks or verification steps, not the primary provisioning pattern.

1. Threat Model & Shared Responsibility

Kubernetes introduces several new high-value control points beyond "just containers": the API server, etcd, admission layer, kubelets, service account tokens, cluster DNS, the CNI plugin, and often the cloud identity bridge. In a managed cluster, the provider operates parts of the control plane, but you still own the workload security, namespace boundaries, RBAC, secrets, and east-west trust model.

Managed Kubernetes is not "managed security". GKE, EKS, and AKS reduce some control-plane operational burden, but they do not remove your responsibility for pod security, RBAC, network policy, secrets handling, image integrity, or workload-to-cloud identity design.

Cluster Trust Boundaries

Humans / CI
   |
   v
Cloud IAM / IdP ------------------------------+
   |                                          |
   v                                          v
Kubernetes API server ---> Admission ---> etcd (desired state, secrets)
        ^                      |
        |                      v
ServiceAccount tokens <--- Pods / Deployments <--- kubelet on each node
        |                      |
        |                      +--> East-west traffic to other pods/services
        |
        +--> External cloud APIs via workload identity

Compromise patterns:
  Pod -> token -> API -> secret -> stronger SA -> node/cloud pivot
  Pod -> privileged escape -> node -> kubelet creds -> cluster pivot
  Pod -> no NetworkPolicy -> lateral movement -> database/internal API

What You Usually Own

Boundary Why it matters Main controls
API server access The API is the cluster's root control plane. If you can create or mutate objects, you can often escalate. Private endpoints, strong auth, audit logs, least-privilege RBAC
Service accounts Pods authenticate as service accounts. Poorly scoped accounts become reusable bearer credentials. Dedicated service accounts, short-lived tokens, automountServiceAccountToken: false by default
Workloads Pods are the attacker foothold most teams actually lose first. Restricted pod security, non-root, no privilege escalation, read-only filesystems
Network Kubernetes is flat by default. Without segmentation, compromise spreads laterally. Default-deny NetworkPolicy, DNS exceptions, explicit ingress/egress allows
Secrets Secrets often contain cloud creds, API keys, database passwords, or service account tokens. Encryption at rest, strict RBAC, external stores, fast rotation
Cloud bridge Cluster compromise often becomes cloud compromise through node roles or workload identity. Workload identity, minimal cloud roles, node metadata hardening

2. Control Plane & API Security

The API server is the most important object in the cluster. It mediates create, update, delete, authentication, authorization, admission, and persistence into etcd. If an attacker can meaningfully interact with the API, they can usually either escalate directly or create a pod that does it for them.

Control Plane Priorities

  • Restrict who can reach the API server, especially on managed clusters where public endpoints are often the default starting point.
  • Use strong external identity and MFA for human admins. Avoid long-lived kubeconfigs shared between people.
  • Turn on audit logging and keep it somewhere attackers in the cluster cannot tamper with.
  • Encrypt sensitive data in etcd, especially Secrets.
  • Keep cluster versions current. Control-plane CVEs are less forgiving than app bugs.
  • Be deliberate about kubelet and node access. Node access is frequently equivalent to "I can become the cluster."

Audit What Matters

Audit logs answer the questions you need during incident response: who created the pod, who changed the role binding, who read the secret, and what service account the pod used. On self-managed Kubernetes, you configure audit policy on kube-apiserver. On managed platforms, you usually enable provider-native control-plane log export.

Example Flow - API Access, Authorization, and Audit
Engineer / CI principal
        |
        v
   kubectl request ------------------------------+
        |                                        |
        v                                        v
  API server ---> authn/authz ---> admission ---> persisted object

  --as=system:serviceaccount:payments:checkout
Common Mistakes
  • Leaving the API server broadly reachable from the internet with only "we trust IAM" as the control.
  • Disabling or never enabling audit logs, which leaves no timeline during investigation.
  • Treating node access as "just infrastructure access" even though node compromise often leads to service account theft and pod exec.
  • For self-managed clusters, skipping admission plugins or encryption-at-rest because the cluster appears to work without them.

3. Workload Hardening

Pods are the part of Kubernetes that attackers are most likely to touch first. A vulnerable app, a leaked credential, or a malicious dependency lands inside a container. Workload hardening is about making that first foothold low-value: hard to escape, hard to abuse, and hard to use for privilege escalation.

Secure Pod Baseline

Risky default Better baseline Why
Run as root runAsNonRoot: true Reduces the blast radius of app compromise and blocks many lazy post-exploitation steps.
Writable root filesystem readOnlyRootFilesystem: true Makes persistence and tool dropper behavior harder.
Default Linux capabilities capabilities.drop: ["ALL"] Shrinks the kernel attack surface.
Privilege escalation allowed allowPrivilegeEscalation: false Stops common escalation chains even after code execution.
No seccomp profile seccompProfile: RuntimeDefault Blocks many unnecessary syscalls by default.
Host namespace access Avoid hostNetwork, hostPID, hostIPC, hostPath These pierce container isolation and frequently turn app compromise into node compromise.

Commented Example: Terraform Deployment with Restricted Pod Settings

Restricted Pod Stack
Container process
    |
    +--> runs as non-root user
    +--> cannot escalate privileges
    +--> has all Linux capabilities dropped
    +--> writes blocked by read-only root filesystem
    +--> seccomp RuntimeDefault narrows syscall surface
    |
    v
Pod boundary stays much harder to abuse after app compromise
  
# Terraform: manage the Deployment as Kubernetes infrastructure.
# The underlying manifest is the same idea as YAML, but Terraform now owns it.
resource "kubernetes_manifest" "payments_api" {
  manifest = {
    apiVersion = "apps/v1"
    kind       = "Deployment"
    metadata = {
      name      = "payments-api"
      namespace = "payments"
    }
    spec = {
      replicas = 2
      selector = {
        matchLabels = {
          app = "payments-api"
        }
      }
      template = {
        metadata = {
          labels = {
            app = "payments-api"
          }
        }
        spec = {
          # Use a dedicated service account rather than "default".
          serviceAccountName            = "payments-api"
          # Do not mount API credentials unless the app actually needs them.
          automountServiceAccountToken  = false
          securityContext = {
            seccompProfile = {
              # Let the runtime apply its maintained syscall filter.
              type = "RuntimeDefault"
            }
          }
          containers = [
            {
              name  = "app"
              # Pin the image by digest so reviewers know exactly what will run.
              image = "ghcr.io/acme/payments-api@sha256:1111111111111111111111111111111111111111111111111111111111111111"
              ports = [
                {
                  containerPort = 8080
                }
              ]
              securityContext = {
                runAsNonRoot             = true
                runAsUser                = 10001
                allowPrivilegeEscalation = false
                readOnlyRootFilesystem   = true
                capabilities = {
                  # Start from zero capabilities and add back only if justified.
                  drop = ["ALL"]
                }
              }
              resources = {
                requests = {
                  cpu    = "100m"
                  memory = "128Mi"
                }
                limits = {
                  cpu    = "500m"
                  memory = "512Mi"
                }
              }
            }
          ]
        }
      }
    }
  }
}
Pod Security Admission is the built-in way to enforce this class of controls. PodSecurityPolicy was removed in Kubernetes v1.25. Today the native baseline is Pod Security Admission plus optional policy layers like Kyverno, Gatekeeper, or built-in ValidatingAdmissionPolicy.

4. RBAC & Service Accounts

RBAC is where many Kubernetes incidents actually become serious. Overly broad permissions do not just mean "a little too much admin access"; they often mean "this principal can mint a pod, mount secrets, impersonate a stronger account, or create a token for another identity."

High-Risk Permissions

Permission or pattern Why it is dangerous
list or watch on secrets Effectively exposes secret contents, not just metadata.
Ability to create pods or deployments Implicitly grants access to mounted Secrets, ConfigMaps, volumes, and any service account in that namespace.
bind, escalate, or impersonate These can convert ordinary access into administrative access very quickly.
create on serviceaccounts/token Lets callers mint credentials for service accounts.
Broad ClusterRoleBinding use Turns namespace-scoped access mistakes into cluster-wide mistakes.

Commented Example: Terraform Least-Privilege RBAC

RBAC Permission Chain
Pod
 |
 v
ServiceAccount "report-reader"
 |
 v
RoleBinding "lease-reader-binding"
 |
 v
Role "lease-reader"
 |
 v
Only get/list/watch on coordination.k8s.io leases in payments namespace
  
# Terraform: give one service account one small role in one namespace.
resource "kubernetes_manifest" "report_reader_sa" {
  manifest = {
    apiVersion = "v1"
    kind       = "ServiceAccount"
    metadata = {
      name      = "report-reader"
      namespace = "payments"
    }
    # Default off. A pod has to opt in to mounting the token.
    automountServiceAccountToken = false
  }
}

resource "kubernetes_manifest" "lease_reader_role" {
  manifest = {
    apiVersion = "rbac.authorization.k8s.io/v1"
    kind       = "Role"
    metadata = {
      name      = "lease-reader"
      namespace = "payments"
    }
    rules = [
      {
        # This rule grants read-only lease access and nothing else.
        apiGroups = ["coordination.k8s.io"]
        resources = ["leases"]
        verbs     = ["get", "list", "watch"]
      }
    ]
  }
}

resource "kubernetes_manifest" "lease_reader_binding" {
  manifest = {
    apiVersion = "rbac.authorization.k8s.io/v1"
    kind       = "RoleBinding"
    metadata = {
      name      = "lease-reader-binding"
      namespace = "payments"
    }
    subjects = [
      {
        kind      = "ServiceAccount"
        name      = "report-reader"
        namespace = "payments"
      }
    ]
    roleRef = {
      # Bind the exact Role above instead of a broad ClusterRole.
      apiGroup = "rbac.authorization.k8s.io"
      kind     = "Role"
      name     = "lease-reader"
    }
  }
}

RBAC Design Rules

  • Prefer RoleBinding over ClusterRoleBinding whenever possible.
  • Give each workload its own service account. Do not reuse one high-privilege service account across unrelated apps.
  • Disable service account token auto-mounting unless the pod really needs Kubernetes API access.
  • Review who can create workloads, create tokens, bind roles, and read secrets. Those are the permissions that usually matter most.

5. Secrets, Encryption & Workload Identity

Many "Kubernetes" breaches are really secret-management breaches wearing a Kubernetes hat. If a pod can read a database password, a cloud access token, or a highly privileged service account token, the attacker rarely cares whether they got there via a container exploit or a sloppy Helm chart.

Base64 is not encryption. Kubernetes Secrets are encoded, not protected, unless you configure encryption at rest for the API server and etcd. The upstream secrets guidance explicitly warns that Secrets are stored unencrypted by default.

What Good Secret Design Looks Like

  • Encrypt Secret data at rest.
  • Minimize who can get, list, or watch Secrets.
  • Prefer short-lived credentials and identity federation over static API keys.
  • Mount only the secret a container needs; do not spray all secrets into environment variables.
  • Use external secret stores for high-value credentials when possible.

Commented Example: Terraform Pod with Projected Service Account Token

Projected Token Flow
Kubernetes ServiceAccount
          |
          v
 short-lived projected JWT
          |
          v
 mounted as a file inside the pod
          |
          +--> Kubernetes API auth
          +--> cloud workload identity exchange

Better than handing every pod a long-lived default token automatically
  
# Terraform: mount a projected service account token only where the workload
# explicitly needs it, instead of accepting the default token mount.
resource "kubernetes_manifest" "token_consumer" {
  manifest = {
    apiVersion = "v1"
    kind       = "Pod"
    metadata = {
      name      = "token-consumer"
      namespace = "payments"
    }
    spec = {
      serviceAccountName           = "report-reader"
      automountServiceAccountToken = false
      containers = [
        {
          name    = "app"
          image   = "busybox:1.36"
          command = ["sleep", "3600"]
          volumeMounts = [
            {
              # Mount the token in an explicit path the app knows to read.
              name      = "k8s-token"
              mountPath = "/var/run/secrets/custom"
              readOnly  = true
            }
          ]
        }
      ]
      volumes = [
        {
          name = "k8s-token"
          projected = {
            sources = [
              {
                serviceAccountToken = {
                  # Kubernetes writes the projected token at this file path.
                  path              = "token"
                  # Audience should match the service that will validate the token.
                  audience          = "kubernetes.default.svc"
                  # Rotate the token frequently instead of relying on a long-lived secret.
                  expirationSeconds = 3600
                }
              }
            ]
          }
        }
      ]
    }
  }
}

6. Network Policies & East-West Traffic

Kubernetes networking starts from a dangerous place: a flat, highly connected environment where pods can usually talk to one another freely. That is convenient for demos and bad for real security. A compromised pod should not be able to probe every internal service, database, metrics endpoint, and admin API by default.

A NetworkPolicy object by itself does nothing unless your network plugin enforces it. Upstream Kubernetes explicitly notes that creating a NetworkPolicy resource without a controller that implements it has no effect.

Commented Example: Terraform Network Policies

Default Deny with Explicit Application Paths
frontend pod  ---- TCP/8080 ----> backend pod ---- TCP/5432 ----> db pod
    |                                  |
    |                                  +--> other destinations denied
    +--> everything else denied

Default deny defines the floor.
Small allow rules define the only legal traffic paths.
  
# Terraform: use separate resources so policy intent stays obvious in plan
# output instead of hiding everything in one giant manifest blob.
resource "kubernetes_manifest" "default_deny_all" {
  manifest = {
    apiVersion = "networking.k8s.io/v1"
    kind       = "NetworkPolicy"
    metadata = {
      name      = "default-deny-all"
      namespace = "payments"
    }
    spec = {
      # Empty selector means "all pods in this namespace".
      podSelector = {}
      # Deny both directions until another policy explicitly allows them.
      policyTypes = ["Ingress", "Egress"]
    }
  }
}

resource "kubernetes_manifest" "allow_frontend_to_backend" {
  manifest = {
    apiVersion = "networking.k8s.io/v1"
    kind       = "NetworkPolicy"
    metadata = {
      name      = "allow-frontend-to-backend"
      namespace = "payments"
    }
    spec = {
      podSelector = {
        matchLabels = {
          app = "backend"
        }
      }
      policyTypes = ["Ingress"]
      ingress = [
        {
          # Only frontend-labeled pods may initiate this path.
          from = [
            {
              podSelector = {
                matchLabels = {
                  app = "frontend"
                }
              }
            }
          ]
          ports = [
            {
              protocol = "TCP"
              port     = 8080
            }
          ]
        }
      ]
    }
  }
}

resource "kubernetes_manifest" "allow_backend_to_db" {
  manifest = {
    apiVersion = "networking.k8s.io/v1"
    kind       = "NetworkPolicy"
    metadata = {
      name      = "allow-backend-to-db"
      namespace = "payments"
    }
    spec = {
      podSelector = {
        matchLabels = {
          app = "backend"
        }
      }
      policyTypes = ["Egress"]
      egress = [
        {
          # Backend may reach only db-labeled pods on PostgreSQL.
          to = [
            {
              podSelector = {
                matchLabels = {
                  app = "db"
                }

DNS Exception under Default Deny
application pods
      |
      +--> UDP/TCP 53 allowed to kube-system DNS
      |
      +--> all other egress still blocked until separately allowed

Without a DNS exception, many "secure" namespaces simply fail to resolve names.
  
# Terraform: add a narrow DNS exception after default-deny egress.
# The kube-dns / CoreDNS labels vary by distribution, so teams often match the
# kube-system namespace first and then refine the pod label in their own cluster.
resource "kubernetes_manifest" "allow_dns_egress" {
  manifest = {
    apiVersion = "networking.k8s.io/v1"
    kind       = "NetworkPolicy"
    metadata = {
      name      = "allow-dns-egress"
      namespace = "payments"
    }
    spec = {
      podSelector = {}
      policyTypes = ["Egress"]
      egress = [
        {
          to = [
            {
              namespaceSelector = {
                matchLabels = {
                  "kubernetes.io/metadata.name" = "kube-system"
                }
              }
            }
          ]
          ports = [
            {
              protocol = "UDP"
              port     = 53
            },
            {
              protocol = "TCP"
              port     = 53
            }
          ]
        }
      ]
    }
  }
}

What NetworkPolicy Does Not Solve

  • It does not replace workload identity or RBAC.
  • It is mainly L3/L4 control; it does not understand business authorization or most HTTP semantics.
  • It does not magically secure hostNetwork pods.
  • It does not secure traffic to cloud APIs unless you define egress policy for those paths too.

Practical Network Segmentation Rules

  • Start every application namespace from deny-all.
  • Add explicit DNS egress allows so workloads can still resolve names.
  • Separate internet egress from east-west policy review; teams often remember one and forget the other.
  • Treat observability, service mesh, and ingress controllers as infrastructure dependencies that need their own documented exceptions.

7. Admission Control & Policy Enforcement

RBAC decides who is allowed to submit a request. Admission decides whether the object itself is acceptable. This is where you block dangerous pod specs, reject mutable image tags, require labels, and enforce org-wide safety rules before the object ever lands in etcd.

Policy Layers

  • Pod Security Admission: built-in, namespace-scoped enforcement of privileged, baseline, and restricted pod security standards.
  • ValidatingAdmissionPolicy: built-in CEL-based validation for richer custom rules.
  • Webhooks / policy engines: Kyverno, Gatekeeper, and cloud policy integrations for broader ecosystems and library-style policies.

Commented Example: Terraform Namespace Labels for Pod Security Admission

Pod Security Admission Namespace Flow
Pod create request
      |
      v
Namespace labels:
  enforce = restricted
  warn    = restricted
  audit   = restricted
      |
      +--> unsafe pod denied
      +--> borderline pod warned/audited
      +--> compliant pod admitted
  
# Terraform-managed namespaces are the cleanest place to enforce Pod Security
# Admission, because the labels are born with the namespace.

    apiVersion = "v1"
    kind       = "Namespace"

      name = "payments"
      labels = {
        "pod-security.kubernetes.io/enforce" = "restricted"
        "pod-security.kubernetes.io/audit"   = "restricted"
        "pod-security.kubernetes.io/warn"    = "restricted"
      }
    }
  }
}

Commented Example: Terraform ValidatingAdmissionPolicy

Validating Admission Policy Decision Path
Deployment create / update
         |
         v
ValidatingAdmissionPolicy
         |
         +--> image ends with ":latest" ? deny
         +--> immutable tag or digest ? allow

Admission moves this check from review culture into enforcement.
  
# Terraform: manage built-in admission resources with kubernetes_manifest.
resource "kubernetes_manifest" "disallow_latest_image_tag" {
  manifest = {
    apiVersion = "admissionregistration.k8s.io/v1"
    kind       = "ValidatingAdmissionPolicy"
    metadata = {
      name = "disallow-latest-image-tag"
    }
    spec = {
      failurePolicy = "Fail"
      matchConstraints = {
        resourceRules = [
          {
            apiGroups   = ["apps"]
            apiVersions = ["v1"]
            operations  = ["CREATE", "UPDATE"]
            resources   = ["deployments"]
          }
        ]
      }
      validations = [
        {
          # CEL evaluates every container in the pod template.
          expression = "object.spec.template.spec.containers.all(c, !c.image.endsWith(':latest'))"
          message    = "Pin images by digest or an immutable tag instead of latest."
        }
      ]
    }
  }
}

resource "kubernetes_manifest" "disallow_latest_image_tag_binding" {
  manifest = {
    apiVersion = "admissionregistration.k8s.io/v1"
    kind       = "ValidatingAdmissionPolicyBinding"
    metadata = {
      name = "disallow-latest-image-tag-binding"
    }
    spec = {
      policyName        = "disallow-latest-image-tag"
      validationActions = ["Deny"]
    }
  }
}
Important limitation: admission control cannot protect reads. If a principal can already get, list, or watch a sensitive object, admission is no longer the relevant control. That is an RBAC and secret-access problem.

8. Supply Chain & Runtime Security

Many clusters are hardened on paper and still deploy vulnerable or untrusted images. Kubernetes security starts before the pod runs: image build provenance, registry controls, vulnerability scanning, immutable references, and policy that prevents "just push it, we will fix it later" from becoming production reality.

Supply Chain Rules

  • Scan images before deploy and block critical issues when policy requires it.
  • Pin production images by digest, not mutable tags.
  • Use private registries and narrow pull credentials.
  • Prefer signed and attested images for higher-trust workloads.
  • Enable AlwaysPullImages in multi-tenant environments where cache reuse can weaken registry authorization assumptions.

Runtime Rules

  • Patch nodes and container runtimes aggressively.
  • Watch for unusual exec, port-forward, crypto-mining, and outbound beaconing behavior.
  • Treat privileged pods, host mounts, and host namespaces as runtime exceptions that need explicit review.
  • Protect kubelet access and node credentials, because node compromise often outranks pod compromise in severity.
# Good image hygiene: use an immutable digest so admission and deploy tools
# can reason about exactly what is running.
containers:
  - name: api

9. Managed Kubernetes: GKE, EKS, and AKS

The right Kubernetes hardening baseline is similar across clouds, but the details are not. Identity, API endpoint exposure, control-plane logging, and policy integrations all vary by provider.

Provider snapshot: The guidance in this section reflects current upstream and provider documentation as of April 3, 2026.
Area GKE EKS AKS
Workload identity Workload Identity Federation for GKE is the recommended path. EKS Pod Identity is the modern AWS-native path; IRSA still exists. Microsoft Entra Workload ID is the preferred identity federation model.
API endpoint posture Private and restricted control-plane connectivity options are available; Autopilot bakes in stronger defaults. Cluster endpoint is public by default; private access should be evaluated early. Private clusters use Private Link so API traffic stays on private networking.

What I Would Prioritize by Platform

GKE

  • Prefer Workload Identity Federation for GKE over older metadata-protection workarounds.
  • Use Autopilot where its workload constraints fit your environment, because Google enforces more of the safe baseline for you.
  • Use Binary Authorization for higher-assurance workloads.

EKS

  • Review API endpoint exposure first, because public endpoint access is the starting default.
  • Enable control-plane logs early instead of waiting for the first incident.
  • Use EKS Pod Identity or IRSA so pods do not inherit the node's AWS permissions.

AKS

  • Use private clusters where your network design supports them.
  • Use Microsoft Entra Workload ID instead of static Azure credentials in Kubernetes Secrets.
  • Use Azure Policy for baseline enforcement and Key Vault CSI for externalized secret access.

10. Real-World Deployment Models for Secure Kubernetes

Organizations do not all "do Kubernetes" the same way. The secure shape of the platform depends on three things: how much the tenants trust each other, how much blast-radius the business is willing to accept, and how much operational overhead the platform team can carry. Those three variables drive almost every deployment model you see in the wild.

Source boundary: the shared-cluster, node-isolation, and virtual-control-plane patterns come directly from upstream Kubernetes and cloud-provider multi-tenancy guidance. The deployment stamp / cell model is an architectural inference that combines those isolation patterns with the cloud deployment-stamp pattern used in large multitenant platforms.

How Platform Teams Usually Decide

Question If the answer is "lower" If the answer is "higher"
How much do tenants trust each other? Shared cluster, namespace-based isolation Dedicated node pools, virtual control planes, or separate clusters
How severe is the regulatory / customer-isolation requirement? Shared infrastructure is usually acceptable Per-customer cluster, dedicated accounts/projects, or stamp/cell isolation
How much platform overhead can the org absorb? Fewer shared clusters More clusters, more automation, more fleet tooling
How much variation exists across workloads? Single platform baseline works well Tiered or hybrid model is usually better

Model 1: Shared Cluster, Namespace per Team or Workload

This is the most common starting point for internal platform teams. One production cluster per region hosts multiple teams or apps. Each tenant gets one or more namespaces, and the platform team uses RBAC, quotas, Pod Security Admission, and NetworkPolicy to keep tenants from stepping on each other.

Why organizations choose it

  • They want the lowest operational overhead.
  • Teams are semi-trusted employees rather than hostile external tenants.
  • Service discovery and platform reuse matter more than hard isolation.
  • The business wants one shared ingress, one shared observability stack, and one shared platform baseline.
Model 1 - Shared Cluster with Namespace Tenants
                 public DNS / WAF / LB
                          |
                          v
                   shared ingress layer
                          |
        +-----------------+-----------------+
        |                                   |
        v                                   v
  namespace team-a                    namespace team-b
    service / pods                      service / pods
    quota / PSA / RBAC                  quota / PSA / RBAC

One control plane, one data plane, many logical tenants.

Step-by-step rollout

  1. Provision one secure cluster per environment and region.
  2. Install the shared edge layer: ingress controller or gateway, cert manager, DNS automation, logging, and metrics.
  3. Create one namespace per team or workload, and label it with Pod Security Admission policy.
  4. Apply RoleBindings, ResourceQuotas, LimitRanges, and default-deny network policies for each namespace.
  5. Let app teams deploy only into their own namespaces through GitOps or Terraform.
  6. Point application hostnames at the shared load balancer and route traffic to the correct namespace service.

Model 2: Shared Cluster with Dedicated Node Pools or Sole-Tenant Nodes

This model keeps the shared control plane but reserves parts of the data plane for specific tenants or sensitivity tiers. Organizations adopt it when one shared cluster is still desirable, but some workloads need stronger separation because of compliance, performance, licensing, or hardware requirements.

Why organizations choose it

  • Some workloads are more sensitive than others, but the org still wants one shared cluster.
  • Teams need GPU, FIPS, confidential, or higher-memory nodes.
  • Noisy-neighbor problems show up in the shared-node model.
  • The business wants a middle ground between one cluster for all and a cluster per tenant.
Model 2 - Shared Control Plane, Segmented Data Plane
                     shared cluster control plane
                                |
     +--------------------------+---------------------------+
     |                          |                           |
     v                          v                           v

 ingress, coredns         standard apps               isolated workloads
 observability            lower trust needs           taints + selectors

Traffic can still enter through one shared edge, or sensitive apps can use
their own internal ingress on the isolated pool.

Step-by-step rollout

  1. Start from a secure shared-cluster baseline.
  2. Create separate node pools for system services, standard apps, and sensitive workloads.
  3. Apply taints and labels to the dedicated pool, and enforce matching tolerations and selectors in policy.
  4. Keep shared ingress on the system pool, or deploy a dedicated ingress for sensitive workloads.
  5. Place higher-risk or regulated apps only on the isolated node pool.
  6. Charge back node-pool cost and track risk tier by namespace or tenant label.

Model 3: Virtual Control Plane per Tenant

This model gives each tenant an isolated Kubernetes API experience while still sharing the underlying worker fleet. Organizations choose it when namespace isolation is not enough, but running full dedicated clusters for every tenant would create too much cost or operational sprawl.

Why organizations choose it

  • Tenants need their own API experience, CRDs, or cluster-scoped objects.
  • The platform wants better control-plane isolation without a full cluster explosion.
  • The business wants strong self-service for tenants or internal platform customers.
  • The team is willing to accept more platform complexity to avoid many physical clusters.
Model 3 - Virtual Control Planes on a Shared Host Cluster
tenant A kubectl ---> virtual control plane A ---+
tenant B kubectl ---> virtual control plane B ---+--> host cluster workers
tenant C kubectl ---> virtual control plane C ---+
                                                  |
                                                  +--> synced workloads on shared nodes

Control-plane isolation improves.
Data-plane isolation still has to be handled separately.

Step-by-step rollout

  1. Provision a secure host cluster that only the platform team administers directly.
  2. Create one host namespace per tenant or business unit.
  3. Deploy a virtual-control-plane product into each host namespace.
  4. Define what gets synchronized into the host cluster, such as workloads, services, or ingress objects.
  5. Keep ingress, network policy, and data-plane isolation enforced in the host cluster.
  6. Expose either a tenant-specific API endpoint or a tenant-specific kubeconfig to the tenant admins.

Model 4: Cluster per Team, Product, or Environment

This is common in larger enterprises. Instead of one giant shared cluster, each team or product line gets its own cluster, usually with a separate cluster for each environment like dev, staging, and prod. Often there is also a separate cloud account, project, or subscription boundary around each cluster.

Why organizations choose it

  • Different teams need independent release cadences and maintenance windows.
  • The platform team wants clearer blast-radius boundaries.
  • The business wants environment isolation and simpler ownership boundaries.
  • Cloud-account or project isolation is already the norm.
Model 4 - Cluster per Team / Environment
payments-prod DNS  ---> payments-prod cluster ---> ingress -> services -> pods
payments-stage ---> payments-stage cluster ---> ingress -> services -> pods
orders-prod   ---> orders-prod cluster   ---> ingress -> services -> pods

Often paired with:
  separate AWS accounts / GCP projects / Azure subscriptions
  centralized logging and policy, decentralized app ownership

Step-by-step rollout

  1. Create the cloud boundary first: AWS account, GCP project, or Azure subscription per team or per environment.
  2. Provision one secure baseline cluster module for each target environment.
  3. Attach each cluster to centralized identity, logging, and policy systems.
  4. Expose only the production cluster publicly if the app is internet-facing; keep lower environments private.
  5. Give the owning team admin rights only in their own clusters or namespaces.
  6. Roll out cluster upgrades progressively by environment instead of all at once.

Model 5: Cluster per Customer or Dedicated Tenant Stack

This is the strongest practical isolation model most SaaS organizations use before going to bespoke infrastructure. One customer or tenant gets its own cluster, often plus a dedicated database and dedicated cloud boundary. The tenant accesses the app through its own hostname, and traffic never shares a Kubernetes control plane with other customers.

Why organizations choose it

  • Regulated or strategic customers require hard isolation.
  • The contract includes customer-managed keys, data locality, or dedicated infrastructure.
  • The business sells a premium or dedicated tier.
  • Risk tolerance for cross-tenant incidents is very low.
Model 5 - Dedicated Cluster per Customer
tenant-a.example.com ---> dedicated edge / LB ---> tenant-a cluster ---> app + db
tenant-b.example.com ---> dedicated edge / LB ---> tenant-b cluster ---> app + db
tenant-c.example.com ---> dedicated edge / LB ---> tenant-c cluster ---> app + db

Every tenant gets a separate cluster boundary.

Step-by-step rollout

  1. Tenant onboarding triggers an IaC pipeline.
  2. The pipeline creates a tenant-specific cloud boundary, cluster, data store, secret store, and DNS name.
  3. Certificates and ingress are issued only for that tenant hostname.
  4. Observability and backup wiring are attached automatically.
  5. Updates are rolled out in rings or maintenance windows tenant by tenant.
  6. When the tenant churns, the whole stack can be archived or destroyed through the same pipeline.

Model 6: Deployment Stamps or Cells

This is what many large SaaS platforms evolve toward. Instead of one giant shared cluster or one cluster per customer, the platform runs multiple repeatable cells or stamps. Each cell contains a cluster and its supporting resources, and each cell serves a bounded subset of tenants. When a cell fills up or a region needs more capacity, the platform creates another one.

Why organizations choose it

  • They need predictable horizontal scaling and bounded blast radius.
  • They want to limit the impact of incidents to one cell instead of the whole platform.
  • They serve many tenants and need a repeatable way to add capacity.
  • The business needs regional placement, deployment rings, or tenant sharding.
Model 6 - Stamp / Cell Routing
                         global router / front door
                                   |

   cluster + ingress + db                          cluster + ingress + db
   tenants A-F mapped here                         tenants G-L mapped here

Add a new cell when capacity, region, or risk rules say to split.

Step-by-step rollout

  1. Define one reusable cell module that includes the cluster, ingress, logging, secrets, and data dependencies.
  2. Deploy one or more cells per region.
  3. Maintain a tenant-to-cell routing map in a central system.
  4. Route requests through a global edge or tenant-specific DNS to the correct cell.
  5. When a cell approaches its limit, deploy a new one and place new tenants there.
  6. Use deployment rings so some cells receive upgrades earlier than others.

Model 7: Hybrid Tiered Platform

Many real organizations end up here. Standard tenants share infrastructure, higher-value tenants get dedicated node pools or dedicated clusters, and premium or regulated customers get dedicated tenant stacks. The routing layer decides which backend pattern a tenant uses.

Why organizations choose it

  • They want one platform with multiple commercial tiers.
  • Most tenants can be low-cost, but a few need stronger isolation.
  • The business wants to upgrade tenants from shared to dedicated without redesigning the whole platform.
  • The platform team needs a migration path instead of a one-time architecture bet.

  1. Build one shared standard cluster as the default landing zone for most tenants.
  2. Add a dedicated-isolation path, either as separate node pools or separate clusters.
  3. Store each tenant's service tier and placement target in a routing map.
  4. Expose a consistent customer hostname pattern regardless of where the tenant runs.
  5. Automate promotion from shared to dedicated placement when contracts, risk, or performance demands change.

How Traffic Is Directed in Real Organizations

In practice, traffic routing is usually one of three things:

  • Shared ingress routing: one load balancer and one ingress or gateway routes by hostname or path to services inside one cluster.
  • Tenant-specific DNS: each tenant hostname points directly to a dedicated cluster or dedicated cell.
  • Global routing layer: a front door, CDN, WAF, or API gateway looks up the tenant placement and forwards traffic to the correct cluster or cell.
Step-by-Step Traffic Resolution
1. Client resolves hostname
2. DNS or global edge sends the request to the chosen cluster or cell
3. External or internal load balancer lands traffic on the cluster edge
4. Ingress / Gateway routes to the correct Service
5. Service forwards to pods
6. NetworkPolicy / service mesh / app auth still decide what is allowed

How to Deploy This in Real Life at an Organization

The implementation order matters. Real platform teams usually decide the isolation model first, then lock in the cloud boundary and traffic path, and only then let application teams onboard. That sequence prevents a shared-cluster prototype from accidentally becoming the long-term production architecture.

1. Classify tenants and workloads

  1. Classify tenants and workloads. Start with a simple placement matrix: trusted internal teams, regulated workloads, premium customers, and standard multitenant workloads. Example: an internal analytics dashboard might fit Model 1, while a dedicated bank customer fits Model 5.
  2. Pick the isolation model before building the cluster. If the business needs hard tenant isolation, decide that now instead of trying to bolt it onto a shared cluster later.
  3. Create the cloud boundary and network path. Example: for Model 4, create one AWS account per environment and peer those VPCs to central logging and identity services. For Model 5, create one Azure subscription or resource group per tenant tier and keep the tenant's ingress, database, and cluster in that boundary.
  4. Provision a reusable cluster baseline module. The module should turn on private API access where possible, logging, secret encryption, workload identity, baseline policy, and shared add-ons such as ingress, cert-manager, DNS automation, and metrics.
  5. Wire the traffic path end to end. Example: `tenant-a.example.com` resolves in DNS, passes through a CDN/WAF, lands on a load balancer, then reaches an ingress controller or Gateway API object that routes to the right service. Internal applications often swap the public edge for private DNS plus VPN, Private Link, peering, or ExpressRoute.
  6. Apply the model-specific placement controls. In Model 1 that means namespaces, RBAC, quotas, and default-deny network policy. In Model 2 it means dedicated node pools plus scheduling policy. In Model 6 it means a tenant-to-cell routing map and DNS or global-edge placement.
  7. Onboard teams or tenants through automation, not tickets. Example: a tenant onboarding pipeline creates the namespace or cluster, issues DNS, attaches logging, and outputs the hostname and deployment repo. That is the difference between a stable platform model and a manual snowflake.
  8. Run it like a fleet. Standardize upgrades, backup policy, cert rotation, vulnerability response, and audit-log export. Shared clusters need strong namespace hygiene; fleet models need strong drift control and routing visibility.
If you choose this model What the real-life setup usually looks like
Model 1 One cluster per environment or region, one shared ingress, one namespace per team, and hostnames routed into namespace-scoped services.
Model 2 One shared cluster with separate node pools for system, standard, and sensitive workloads, plus taints and policy enforcing which apps land where.
Model 4 A cluster module deployed many times, often one per team and environment, with private control planes and public application ingress only where needed.
Model 5 A tenant onboarding pipeline creates a dedicated cluster, ingress hostname, secrets wiring, database, and observability stack for each customer.
Model 6 A global router or DNS layer maps each tenant to a specific cell, and the platform adds new cells as capacity or risk boundaries require.

End-to-End Example: One Org, Many Sites and Apps

This walkthrough shows how a single organization can mix the models above to run a marketing site, a customer portal, and internal ops tools without building a one-off platform for each.

Traffic and Placement Overview
               Public DNS / CDN / WAF
                     |
                     +--> www.example.com -> shared-web cluster (Model 1)
                     |
                     +--> portal.example.com -> dedicated cluster (Model 5)

         Internal DNS / VPN / Private Link
                     |
                     +--> ops.internal.example.com -> private ops cluster (Model 4)

Data boundaries:
- portal runs in its own cloud account/project + cluster
- ops cluster is private-only; API endpoint and ingress stay on private networking
- shared web apps reuse one shared edge and shared cluster

Step-by-step rollout

  1. Create three cloud boundaries: web (shared), portal (dedicated), ops (private internal).
  2. Provision clusters:
    • shared-web: Model 1 (namespaces per app) with public ingress.
    • portal-prod: Model 5 (dedicated cluster) with private API, public app ingress only.
    • ops-internal: Model 4 (cluster per environment) with private API and internal ingress only.
  3. Apply placement policy:
    • marketing and docs namespaces on shared-web with default-deny policy.
    • portal namespace only on portal-prod, backed by dedicated DB and secret store.
    • ops namespace on ops-internal with VPN/Private Link–only ingress.
  4. Wire traffic:
    • Public DNS: www.example.com CNAME -> shared-web ingress hostname.
    • Public DNS: portal.example.com CNAME -> portal-prod ingress hostname.
    • Internal DNS: ops.internal.example.com -> internal load balancer IP in ops-internal.
  5. Enforce identity and policy:
    • PSA restricted on all namespaces; admission policy blocks privileged/hostPath everywhere.
    • Workload identity per cluster (GKE Workload Identity, EKS Pod Identity, AKS Workload ID).
    • Default-deny egress; explicit allow for DNS, database, and required third-party APIs per namespace.

Terraform sketch (DNS + placement)

# Assume cluster modules already output ingress hostnames.
locals {
  sites = {
    "www"    = { cluster = "shared-web",   hostname = "www.example.com" }
    "docs"   = { cluster = "shared-web",   hostname = "docs.example.com" }
    "portal" = { cluster = "portal-prod",  hostname = "portal.example.com" }
  }
}

# Route53 example for public sites; use your provider equivalents.
resource "aws_route53_record" "site_dns" {
  for_each = local.sites

  zone_id = aws_route53_zone.public.zone_id
  name    = each.value.hostname
  type    = "CNAME"
  ttl     = 300

  records = [
    module.cluster[each.value.cluster].ingress_hostname
  ]
}

# Namespace and NetworkPolicy for the shared web cluster.
resource "kubernetes_manifest" "marketing_namespace" {
  manifest = {
    apiVersion = "v1"
    kind       = "Namespace"
    metadata = {
      name = "marketing"
      labels = { "pod-security.kubernetes.io/enforce" = "restricted" }
    }
  }
}

resource "kubernetes_manifest" "marketing_default_deny" {
  manifest = {
    apiVersion = "networking.k8s.io/v1"
    kind       = "NetworkPolicy"
    metadata = { name = "default-deny", namespace = "marketing" }
    spec = {
      podSelector = {}
      policyTypes = ["Ingress", "Egress"]
      egress = [
        {
          to = [
            { namespaceSelector = { matchLabels = { "kubernetes.io/metadata.name" = "kube-system" } } }
          ]
          ports = [
            { protocol = "UDP", port = 53 }
          ]
        }
      ]
    }
  }
}

resource "aws_route53_record" "ops_internal" {

  ttl     = 60
  records = [module.cluster["ops-internal"].internal_ingress_ip]
}

What Most Organizations Actually Do Over Time

Many teams start with Model 1, then add Model 2 for sensitive workloads. As the business grows, they move some domains into Model 4 or Model 6. Regulated or premium customers often get Model 5 or the premium side of Model 7. In other words: Kubernetes platforms usually become more mixed over time, not less.

11. Attack Paths & Pentesting Mindset

The most useful way to assess Kubernetes security is to think in chains, not isolated findings. An exposed dashboard is bad, but a dashboard that lets you create a privileged pod with the default service account in a namespace with no network policy is a full story. That is what matters.

Common Attack Chains

Chain 1: App RCE -> SA token -> API reads -> secret theft
  Vulnerable app
    -> read mounted service account token
    -> call API server
    -> list pods/configmaps or read secrets
    -> steal DB or cloud credentials

Chain 2: App RCE -> privileged pod -> node compromise
  Pod creation rights or privileged pod already exists
    -> mount hostPath or use host namespaces
    -> escape to node
    -> access kubelet / runtime / other pods
    -> pivot cluster-wide

Chain 3: SSRF -> metadata -> cloud pivot
  Workload reaches node metadata or cloud credential endpoint
    -> retrieve node-scoped credentials
    -> access cloud APIs
    -> read storage, secrets, registries, or create new compute

Chain 4: Flat network -> lateral movement
  One pod compromised
    -> scan internal services
    -> reach admin UIs, metrics, Redis, Elasticsearch, or databases
    -> expand access without touching Kubernetes RBAC at all
What Attackers Look For First
  • Mounted service account tokens
  • Pods with privileged: true or host mounts
  • Ability to create workloads in any namespace
  • Readable Secrets and overly broad ConfigMaps
  • No network policy or trivial east-west access
  • Cloud metadata access from pods
  • Default service accounts reused everywhere

Hands-On Attack and Defence Examples (Lab Focus)

Scenario 1 - Steal Secrets via Service Account Token
Attacker path (lab):
1) Compromise app pod
2) Read mounted SA token
3) Call API server and list secrets

How to do it (lab):
kubectl exec -n app ns-pod -- \
  sh -c "APISERVER=https://$KUBERNETES_SERVICE_HOST:$KUBERNETES_SERVICE_PORT; \
         TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token); \
         curl -ks $APISERVER/api/v1/secrets -H \"Authorization: Bearer $TOKEN\""

Prevent:
- Disable automount for service accounts by default
- Use projected, short-lived tokens
- Lock RBAC: most pods should not list or get secrets
- NetworkPolicy so pods cannot reach the API server if they do not need to
Scenario 2 - Privileged Pod + hostPath = Node Escape
Attacker path (lab):
1) Get workload creation rights
2) Deploy privileged pod with hostPath /etc
3) Read node secrets or host SSH keys

How to do it (lab YAML):
apiVersion: v1
kind: Pod
metadata: {name: escape, namespace: lab}
spec:
  hostPID: true
  containers:
  - name: sh
    image: alpine
    securityContext:
      privileged: true
    volumeMounts:
    - {name: hostetc, mountPath: /host/etc}
    command: ["sh","-c","ls /host/etc && cat /host/etc/hosts"]
  volumes:
  - name: hostetc
    hostPath: {path: /etc}

Prevent:
- Pod Security Admission at restricted level (blocks privileged + hostPath)
- Mutating/ValidatingAdmissionPolicy to refuse hostPath/privileged except for approved namespaces
- Separate node pools for system daemons; no app-write to those nodes
Scenario 3 - No NetworkPolicy, Easy Lateral Movement
Attacker path (lab):
1) Get shell in any pod
2) Scan cluster services and reach databases directly

How to do it (lab):
kubectl exec -n app ns-pod -- sh -c "apk add --no-cache nmap && nmap -sT 10.0.0.0/16"
kubectl exec -n app ns-pod -- sh -c "nc -vz postgres.internal 5432"

Prevent:
- Default-deny ingress and egress NetworkPolicy in every namespace
- Allow only required destinations (e.g., app -> DB) and DNS egress
- Use a mesh or gateway for east-west auth if policy alone is not enough
Scenario 4 - Public API Endpoint Exposure
Attacker path (lab):
1) API server left public
2) Stolen cloud creds or mis-scoped IAM lets attacker call the API
3) Create privileged pods, exfiltrate data

How to check (EKS example):
aws eks describe-cluster --name prod --query "cluster.resourcesVpcConfig.endpointPublicAccess"

Prevent:
- Prefer private API endpoints; restrict public access with CIDR allowlists if you must keep it on
- MFA + SSO for admins; short-lived credentials only
- Audit logs on; alert on CreatePod/CreateClusterRoleBinding in prod

Break the Chain Early

  • Even if the app gets RCE, do not hand it API credentials by default.
  • Even if the attacker gets API access, do not give that identity secret-read or workload-creation rights.
  • Even if the attacker lands in one pod, do not let it talk to everything else.
  • Even if one node is lost, do not let that node identity access every cloud resource.

12. Hardening Checklist

If you only implement a handful of Kubernetes controls, make them the ones that most consistently break real attack chains.

Priority What to do Why first
Now Enable Pod Security Admission with restricted for most namespaces Removes a large class of dangerous pod specs quickly
Now Review RBAC for workload creation, secret access, token creation, and broad cluster bindings These permissions drive most escalations
Now Disable service account token auto-mounting by default Stops many pod-to-API pivots
Now Turn on audit logging and protect the log destination You need a timeline when things go wrong
Now Encrypt Secrets at rest Reduces etcd and control-plane disclosure risk
Soon Adopt default-deny ingress and egress network policy Reduces lateral movement and exfiltration paths
Soon Move workloads to cloud workload identity instead of static cloud keys Replaces long-lived credentials with short-lived federated identity
Soon Pin images by digest and enforce policy against mutable tags Makes runtime state auditable and repeatable
Advanced Add validating admission policies for org-specific guardrails Prevents drift and turns security rules into code
Advanced Layer in signing, provenance, and runtime detection Improves trust in what was deployed and catches post-deploy abuse
Simple mental model: make every pod low privilege, make every identity narrow, make every network path explicit, and make every high-risk change visible. If you do those four things well, Kubernetes becomes much harder to abuse.
