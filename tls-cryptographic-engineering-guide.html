TLS: The Complete Picture

From handshake bytes to cryptographic engineering career

Transport Layer Security (TLS) sits between the transport layer (TCP) and the application layer (HTTP, SMTP, etc.). Its job is to provide three guarantees: confidentiality (no one can read the data), integrity (no one can tamper with it undetected), and authentication (you're talking to who you think you are).

TLS 1.3 (RFC 8446, finalized 2018) is the current standard and represents a major cleanup over TLS 1.2. It removed insecure algorithms, reduced the handshake from 2 round-trips to 1, and eliminated entire attack classes by design. If you're building anything today, TLS 1.3 is the baseline.

Why TLS 1.3 matters: It removed RSA key transport (which had no forward secrecy), static DH, CBC-mode ciphers, RC4, 3DES, MD5, SHA-1, compression, and renegotiation. Every removed feature had a named attack against it.

The TLS 1.3 Handshake

The TLS 1.3 handshake completes in a single round-trip (1-RTT). The client speculatively sends key shares in the first message, assuming which key exchange groups the server will accept. Here's the full flow:

The {curly braces} notation means those messages are encrypted. In TLS 1.3, everything after ServerHello is encrypted — including the certificate. This is a privacy improvement over TLS 1.2, where the cert was sent in plaintext.
0-RTT Resumption (PSK)

If the client has previously connected to the server and received a session ticket, it can send encrypted application data in the very first message (0-RTT). The client uses the pre-shared key (PSK) from the previous session to encrypt the early data.

The catch: 0-RTT data is not forward-secret against replay attacks. An attacker who records the ClientHello + early data can replay it. The server must implement anti-replay measures (e.g., single-use tickets, strike registers, or time windows). Because of this, 0-RTT should only carry idempotent requests (like GET). This is a deliberate tradeoff of security for performance.

ClientHello + early_data + key_share + psk_identity
  → [0-RTT application data, encrypted with PSK-derived key]

ServerHello + key_share + pre_shared_key
  → {EncryptedExtensions}
  → {Finished}

  → Full forward secrecy restored from here on
Key Derivation: The HKDF Schedule

TLS 1.3 uses HKDF (HMAC-based Key Derivation Function, RFC 5869) with a structured key schedule. This is one of the most elegant parts of the protocol:

              |

              |

              |

Every derived key incorporates the transcript hash up to that point. This means that if any handshake message is tampered with, the derived keys won't match, and the Finished verification will fail. It's authentication baked into the key schedule itself.

Cipher Suites in TLS 1.3

TLS 1.3 dramatically simplified cipher suite naming. In TLS 1.2, a cipher suite encoded four choices (key exchange, authentication, encryption, hash). In TLS 1.3, key exchange and authentication are negotiated separately via extensions, so the cipher suite only names the AEAD algorithm and hash:

Key Exchange: How ECDHE Works

Every TLS 1.3 connection uses ephemeral Diffie-Hellman for key exchange (either ECDHE or FFDHE, but ECDHE over curve X25519 or P-256 is dominant). "Ephemeral" means fresh key pairs are generated for every connection — this is what gives TLS 1.3 forward secrecy.

The ECDHE Dance

1. Client generates ephemeral key pair:
   private_c = random scalar
   public_c  = private_c × G    (G is the curve's generator point)

2. Server generates ephemeral key pair:
   private_s = random scalar
   public_s  = private_s × G

3. Client sends public_c in ClientHello key_share
   Server sends public_s in ServerHello key_share

4. Both sides compute the shared secret:
   Client:  shared = private_c × public_s = private_c × (private_s × G)
   Server:  shared = private_s × public_c = private_s × (private_c × G)

   Both equal:  private_c × private_s × G
Forward secrecy: Once the handshake is done, both sides discard the ephemeral private keys. Even if the server's long-term private key (the one in its certificate) is later compromised, past session keys cannot be recovered — because they depended on ephemeral keys that no longer exist.

In TLS 1.2 with RSA key exchange (now removed), the client encrypted a random pre-master secret with the server's RSA public key. If that RSA key was ever compromised, every past session could be decrypted. This is exactly why TLS 1.3 mandates ephemeral DH.

Certificate Verification: What the Client Actually Does

When the client receives the server's certificate chain during the handshake, it runs through this verification sequence. Failure at any step = connection rejected.

Step 1: Chain Building

The server sends its leaf cert plus any intermediate certs. The client must build a path from the leaf up to a trusted root in its local store. Sometimes the server misconfigures and omits intermediates — the client may attempt to fetch missing intermediates via the Authority Information Access (AIA) extension, but this is unreliable and slow.

Step 2: Signature Verification (walking up)

For each link in the chain: take the issuer's public key and verify the signature on the issued cert. Concretely, the client hashes the cert's TBS (To Be Signed) data and checks the signature against the issuer's public key. If the math doesn't check out, the cert has been tampered with or wasn't really signed by that issuer.

Step 3: Validity Period

Check that the current time falls within each cert's notBefore and notAfter window. Certificates currently max out at 398 days for public TLS (Apple/Google enforce this), and the CA/Browser Forum approved ballot SC-081 (March 2025) to progressively reduce maximum lifetime: 200 days by March 2026, 100 days by March 2027, and 47 days by March 2029.

Step 4: Revocation Check

Check whether any cert in the chain has been revoked, via CRL or OCSP (discussed below).

Step 5: Name Matching

Verify that the hostname the client requested matches a dNSName in the Subject Alternative Name (SAN) extension. The old Common Name (CN) field is deprecated for this purpose. Wildcards (*.example.com) are allowed but only for a single label — *.example.com matches foo.example.com but not foo.bar.example.com.

Step 6: Constraint Checks

Verify Basic Constraints (a leaf cert must have CA:FALSE, intermediates need CA:TRUE), Key Usage, Extended Key Usage (must include id-kp-serverAuth for TLS), and path length constraints.

Step 7: CertificateVerify

This is separate from chain validation — it's the server proving it actually possesses the private key corresponding to the public key in the cert. The server signs a hash of the entire handshake transcript. Without this, someone could serve a valid cert they don't own the key for (e.g., a CDN serving a cached cert).

Lab Example — Bash / OpenSSL: Walk a Real Certificate Verification Flow 
HOST=example.com
CHAIN_FILE="/tmp/${HOST}-chain.txt"
LEAF_FILE="/tmp/${HOST}-leaf.pem"

# 1. Fetch the live certificate chain from the server.
#    -servername sends SNI so the server returns the correct leaf cert.
printf '' | openssl s_client \
  -connect "${HOST}:443" \
  -servername "${HOST}" \
  -showcerts > "${CHAIN_FILE}"

# 2. Extract only the first certificate block from the chain.
#    This is typically the leaf certificate for the hostname we requested.
awk '
  /BEGIN CERTIFICATE/ { printing = 1 }
  printing { print }
  /END CERTIFICATE/ { exit }
' "${CHAIN_FILE}" > "${LEAF_FILE}"

# 3. Inspect the fields a client cares about during validation.
#    - subject / issuer: who this cert is for and who signed it
#    - dates: validity window
#    - SAN: the real hostname list used for name matching
#    - EKU/basic constraints: is this allowed to be a TLS server cert?
openssl x509 -in "${LEAF_FILE}" -noout \
  -subject \
  -issuer \
  -dates \
  -ext subjectAltName \
  -ext extendedKeyUsage \
  -ext basicConstraints

# 4. Ask OpenSSL to enforce hostname verification during the handshake.
#    If the name does not match, this fails even if the cert is otherwise valid.
printf '' | openssl s_client \
  -connect "${HOST}:443" \
  -servername "${HOST}" \
  -verify_hostname "${HOST}" \
  -brief

This lab maps directly to the checklist above: build the chain, inspect issuer and SAN, verify dates and key usage, then enforce hostname matching during the handshake.

Lab Example — Python: Fail Closed on Bad Chains or Hostnames 
# verify_server_cert.py
# Standard-library example: connect like a real HTTPS client and let
# the OS trust store + Python's ssl module enforce PKI rules.

import pprint
import socket
import ssl

HOST = "example.com"
PORT = 443

# Load the platform trust store (system roots).
ctx = ssl.create_default_context()

# Refuse connections unless the server cert chains to a trusted root.
ctx.verify_mode = ssl.CERT_REQUIRED

# Also require the requested hostname to match the SAN on the leaf cert.
ctx.check_hostname = True

with socket.create_connection((HOST, PORT), timeout=5) as tcp_sock:
    # server_hostname is critical:
    # - sends SNI so the server knows which cert to return
    # - gives Python the hostname to verify against SAN entries
    with ctx.wrap_socket(tcp_sock, server_hostname=HOST) as tls_sock:
        print("Negotiated TLS version:", tls_sock.version())
        print("Cipher suite:", tls_sock.cipher())

        # If we got here, chain validation + hostname verification already passed.
        print("Validated peer certificate:")
        pprint.pprint(tls_sock.getpeercert())

If the certificate is expired, signed by an unknown CA, or does not match HOST, Python raises ssl.SSLError during the handshake instead of continuing with an unsafe connection.

SNI & Encrypted Client Hello

The SNI Problem

Server Name Indication is a plaintext field in the ClientHello that tells the server which hostname the client wants. This is necessary because multiple domains can share an IP address and the server needs to present the right certificate before encryption begins.

But "plaintext" means anyone on the network path can see exactly which site you're connecting to — ISPs, firewalls, nation-state censors. Even with HTTPS everywhere, SNI leaks your browsing destinations.

Encrypted Client Hello (ECH)

ECH (the successor to ESNI) hides the real ClientHello inside an encrypted inner ClientHello. The network still sees an outer ClientHello so routing can happen, but privacy-sensitive fields from the real handshake, including the true SNI, are carried in the encrypted inner one. In practice, this is usually deployed with a client-facing server such as a CDN that publishes ECH configuration via HTTPS/SVCB DNS records.

ECH Request Path
🌐
1. Browser gets HTTPS record + ECH config
Client resolves the site's HTTPS/SVCB DNS record and learns the public-facing name and ECH configuration published for that deployment.
🔒
2. Browser sends an outer ClientHello
The visible handshake targets the client-facing server. An observer can still see an outer hello, but not the real destination hostname carried inside the encrypted inner hello.
☁
3. CDN / client-facing server decrypts the inner hello
Using the published ECH configuration, the edge decrypts the inner ClientHello, learns the real SNI, and selects the correct backend certificate and origin route.
🛡
4. TLS continues with the real backend identity
The origin still proves possession of the correct private key through the normal TLS handshake. ECH hides metadata; it does not remove certificate validation.
Concrete Example — What the Network Sees vs. What the CDN Sees 
Goal: reach https://secret-site.example.com through a CDN.

1. Browser resolves HTTPS records and learns an ECH config.
2. Browser sends:

   Outer ClientHello (visible on path)
     public_name = "cdn.example.net"
     supported_versions = TLS 1.3
     alpn = ["h2", "h3"]

   Inner ClientHello (encrypted)
     server_name = "secret-site.example.com"
     alpn = ["h2"]
     key_share = X25519(...)

3. Observer on the network:
   Sees only that you are connecting to the CDN's public-facing name.

4. Client-facing server:
   Decrypts the inner hello and routes the request to secret-site.example.com.
ECH deployment status: Still conditional rather than universal. Cloudflare supports ECH on its edge, and Chrome's current enterprise documentation says actual use depends on server support, availability of the HTTPS DNS record, and rollout status. In practice, secure DNS matters: if a network attacker can tamper with DNS, they can interfere with ECH discovery and force fallback.

TLS Inspection (Break & Inspect)

A corporate proxy performs a sanctioned man-in-the-middle by maintaining two separate TLS sessions:

TLS Inspection: Two Separate TLS Sessions
Client
(browser)
⟵ TLS Session 1 ⟶
Cert signed by
Corporate CA
Proxy
(Zscaler, etc.)
Decrypts → Inspects → Re-encrypts
⟵ TLS Session 2 ⟶
Real cert from
public CA
Origin Server
(example.com)

This works because the organization pushes their internal root CA cert into every managed device's trust store. The proxy dynamically generates leaf certs for each destination, signed by this internal CA. The browser chains them to the internal root and trusts them.

This breaks the end-to-end trust model. The proxy sees all traffic in plaintext — passwords, tokens, personal data. It also means the proxy must properly validate the server's real cert on Session 2, or it could silently accept a compromised server cert and tunnel the traffic through.

What the Proxy Actually Does

The proxy is not "cracking" a single end-to-end TLS tunnel. It is acting as the TLS endpoint on both sides. That means it terminates one TLS session from the browser, obtains the plaintext HTTP data inside the proxy, applies policy to that plaintext, then creates a completely separate TLS session to the real origin and sends the data onward.

Break & Inspect Request Path
1
Browser requests https://example.com
On a managed network, the browser's traffic is routed through the inspection proxy instead of going straight to the origin server.
2
Proxy presents a cert for example.com signed by the corporate CA
Because the device already trusts the organization's root CA, the browser accepts this proxy-issued leaf certificate as valid for example.com.
3
Browser and proxy derive TLS session keys for Session 1
From the browser's point of view, the proxy is the HTTPS server. TLS records are encrypted on the wire, but the proxy can decrypt them because it is one endpoint of that session.
4
Proxy decrypts, inspects, logs, or blocks the plaintext
Inside the proxy process, the HTTP request exists in plaintext in memory after TLS record decryption. This is where DLP, malware scanning, URL categorization, and policy enforcement happen.
5
Proxy opens a second TLS session to the real origin
Now the proxy becomes the client. It validates the origin's real certificate, negotiates a second set of TLS keys, and re-encrypts the request for the real server.
6
Response takes the reverse path
The origin's response is decrypted inside the proxy, optionally inspected again, and then re-encrypted back to the browser using the browser-proxy session keys.

Why the Browser Trusts the Proxy's Certificate

Managed device trust store:
  Corporate Root CA   ← installed by MDM / Group Policy / endpoint management

Proxy presents:
  Leaf certificate for example.com
  Issuer = Corporate Root CA (or a corporate intermediate)

Browser builds chain:
  example.com leaf → corporate intermediate/root → trusted local root store

Result:
  Browser accepts the cert because:
    1. Hostname matches example.com
    2. Signature chains to a locally trusted CA
    3. Validity / EKU checks pass

This is why the lock icon can still appear in the browser even though the browser is no longer authenticating the public certificate that example.com would normally present on the open internet.

What "Decrypts and Re-encrypts" Means

The traffic is still encrypted on both network hops. Plaintext exists only inside the proxy after one TLS session is terminated and before the bytes are written into the second TLS session.

Browser
  -- TLS records encrypted with key set A -->
Proxy
  -- decrypt with key set A -->
  HTTP plaintext in proxy memory
  -- inspect / log / block / modify -->
  -- encrypt with key set B -->
Origin server

So the proxy is not "forwarding ciphertext unchanged." It is translating between two independent cryptographic channels:

Session 1: Browser  ⇄  Proxy   (proxy acts as TLS server)
Session 2: Proxy    ⇄  Origin  (proxy acts as TLS client)
Why Session 2 validation matters: The browser can only validate Session 1. It has no direct visibility into whether the proxy safely validated the real origin certificate on Session 2. If the proxy skips hostname checks, trusts the wrong roots, or accepts a weak / malicious cert upstream, the browser may still show a trusted connection because it only sees the corporate-issued certificate.
Concrete Failure Mode — How a Bad Upstream Validation Policy Hides Risk 
1. Browser connects to proxy for https://example.com
2. Proxy presents a corporate-issued cert for example.com
3. Browser trusts it because corporate root CA is installed locally
4. Proxy connects upstream to "example.com"
5. If the proxy wrongly accepts:
     - a cert for the wrong hostname
     - a cert signed by an unapproved CA
     - an expired cert
     - a malicious cert from an attacker on the path
6. Browser does not see that mistake directly
7. Browser still shows a trusted Session 1 to the proxy

So the trust boundary has moved:
  from "browser authenticates origin directly"
  to   "browser authenticates proxy, and proxy authenticates origin"

Lab Example — Python: Observe Plaintext After TLS Termination 
# lab_tls_terminator.py
# For a local lab only. Assumes you already have a lab cert/key pair.

import socket
import ssl

LISTEN_HOST = "127.0.0.1"
LISTEN_PORT = 8443
CERT_FILE = "lab-proxy-cert.pem"

    listener.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)

    listener.listen(5)
    print(f"Listening on https://{LISTEN_HOST}:{LISTEN_PORT}")

    client_tcp, addr = listener.accept()
    print("TCP client:", addr)

    with client_tcp:
        with server_ctx.wrap_socket(client_tcp, server_side=True) as browser_tls:

            request_bytes = browser_tls.recv(8192)

            print("----- Plaintext visible inside proxy after decryption -----")
            print(request_bytes.decode("iso-8859-1", errors="replace"))

            response = (

Lab Example — Python: Validate Upstream Cert and Re-encrypt to the Origin 
# lab_upstream_tls.py
# This models the proxy's "Session 2" leg to the real server.

import socket
import ssl
from pprint import pprint

HOST = "example.com"
PORT = 443

request_bytes = (
    f"GET / HTTP/1.1\r\n"
    f"Host: {HOST}\r\n"
    "Connection: close\r\n"
    "\r\n"
).encode("ascii")

client_ctx = ssl.create_default_context()
client_ctx.minimum_version = ssl.TLSVersion.TLSv1_2
client_ctx.check_hostname = True
client_ctx.verify_mode = ssl.CERT_REQUIRED

with socket.create_connection((HOST, PORT), timeout=5) as upstream_tcp:
    with client_ctx.wrap_socket(upstream_tcp, server_hostname=HOST) as origin_tls:
        print("Session 2 established:", origin_tls.version())
        print("Cipher suite:", origin_tls.cipher())
        print("Validated peer certificate:")
        pprint(origin_tls.getpeercert())

        origin_tls.sendall(request_bytes)

        chunks = []
        while True:
            data = origin_tls.recv(4096)
            if not data:
                break
            chunks.append(data)

response = b"".join(chunks)
print(response.decode("iso-8859-1", errors="replace"))

A real inspection proxy combines the two patterns: terminate TLS from the browser, inspect the plaintext in memory, then create a separate outbound TLS client connection like this one and forward the approved request. Production systems also need streaming I/O, HTTP CONNECT handling, ALPN, HTTP/2 or HTTP/3 awareness, logging controls, timeout handling, and very strict upstream certificate validation.

Revocation & Certificate Transparency

CRL (Certificate Revocation Lists)

The CA periodically publishes a signed list of revoked certificate serial numbers. Clients download and cache it. Problems: lists can get huge, they're updated infrequently (hours to days), and clients may fail-open if the CRL is unavailable.

OCSP (Online Certificate Status Protocol)

Client sends the cert's serial number to the CA's OCSP responder and gets a signed "good/revoked/unknown" response. Better than CRL for freshness, but it adds latency (extra round trip) and has a privacy problem — the CA learns which sites you're visiting.

OCSP Stapling

The server pre-fetches its own OCSP response from the CA, caches it, and includes it in the TLS handshake (in the Certificate message via the status_request extension). The client gets a fresh, signed revocation status without contacting the CA. This is the recommended approach, but the server must implement it.

Certificate Transparency (CT)

CT doesn't prevent mis-issuance — it makes it detectable. Every publicly trusted CA must submit every cert it issues to multiple append-only CT logs before the cert is usable. The cert includes Signed Certificate Timestamps (SCTs) proving log inclusion. Browsers (Chrome especially) require SCTs and will reject certs without them.

This means if a CA is compromised and issues a fraudulent cert for your domain, it will appear in CT logs, and monitoring services can alert you. Google's approach after DigiNotar was: "We can't prevent CAs from being compromised, but we can make it impossible to hide."

Lab Example — Bash / OpenSSL: Check for OCSP Stapling and CT Hints 
HOST=example.com

# 1. Ask the server to staple an OCSP response into the handshake.
#    If stapling is configured, OpenSSL prints the stapled response details.
printf '' | openssl s_client \
  -connect "${HOST}:443" \
  -servername "${HOST}" \
  -status

# 2. Pull the live leaf certificate and inspect where revocation info lives.
#    -ocsp_uri shows the responder the client would contact if no staple is present.
printf '' | openssl s_client \
  -connect "${HOST}:443" \
  -servername "${HOST}" 2>/dev/null | \
  openssl x509 -noout -ocsp_uri -text | \
  sed -n '/Authority Information Access/,+6p'

# 3. Look for embedded Signed Certificate Timestamps (SCTs) in the cert.
#    Many public TLS certs include an SCT extension for CT compliance.
printf '' | openssl s_client \
  -connect "${HOST}:443" \
  -servername "${HOST}" 2>/dev/null | \
  openssl x509 -noout -text | \
  sed -n '/CT Precertificate SCTs/,+12p'

This gives you the practical signals behind the theory: whether stapling is active, where OCSP would be queried, and whether CT proof is embedded in the presented certificate.

Certificate Pinning

Certificate pinning is a technique where an application hardcodes (or "pins") the identity of the expected certificate or public key, rather than relying solely on the PKI chain of trust. Even if a rogue CA issues a valid certificate for your domain, a pinning client will reject it because the pin doesn't match.

Types of Pinning

1. Public Key Pinning (pin the SPKI hash)

The most flexible approach. You pin a SHA-256 hash of the Subject Public Key Info (SPKI) — the raw public key bytes. This survives certificate renewals as long as you reuse the same key pair. If you rotate keys, you need backup pins.

# Extract the SPKI pin from a certificate:
openssl x509 -in cert.pem -pubkey -noout | \
  openssl pkey -pubin -outform DER | \
  openssl dgst -sha256 -binary | \
  base64

# Result: "YLh1dUR9y6Kja30RrAn7JKnbQG/uEtLMkBgFF2Fuihg="

2. Certificate Pinning (pin the entire cert)

Pin the hash of the full X.509 certificate. Simpler but more brittle — every certificate renewal breaks the pin, even if the key stays the same. Generally not recommended for production.

3. CA Pinning (pin the issuing CA)

Instead of pinning the leaf, pin the intermediate or root CA. Less restrictive: any cert from that CA will pass. Used when you control which CA issues your certs (e.g., you always use DigiCert) but want to reject certs from other CAs. This is effectively what Certificate Transparency achieves at a broader level.

Lab Example — Bash + Python: Compute a Pin and Compare It to an Expected Value 
# Bash: derive the SPKI pin from a PEM certificate
CERT="cert.pem"

# 1. Extract the public key from the certificate.
# 2. Convert it to DER so we hash the canonical binary SPKI bytes.
# 3. SHA-256 hash the SPKI.
# 4. Base64-encode the digest in the format pinning libraries expect.
openssl x509 -in "${CERT}" -pubkey -noout | \
  openssl pkey -pubin -outform DER | \
  openssl dgst -sha256 -binary | \
  base64

# compute_pin.py
# Python example that shells out to OpenSSL for X.509 parsing, then compares
# the computed SPKI pin to the pin your app expects.

import base64
import hashlib
import subprocess

CERT_FILE = "cert.pem"
EXPECTED_PIN = "YLh1dUR9y6Kja30RrAn7JKnbQG/uEtLMkBgFF2Fuihg="

# Extract the PEM-encoded public key from the certificate.
pubkey_pem = subprocess.check_output(
    ["openssl", "x509", "-in", CERT_FILE, "-pubkey", "-noout"]
)

# Convert the public key to DER so the hash is stable and canonical.
pubkey_der = subprocess.check_output(
    ["openssl", "pkey", "-pubin", "-outform", "DER"],
    input=pubkey_pem,
)

# Compute the SPKI pin exactly the way mobile pinning frameworks do.
actual_pin = base64.b64encode(hashlib.sha256(pubkey_der).digest()).decode("ascii")

print("Computed pin:", actual_pin)
print("Expected pin:", EXPECTED_PIN)

if actual_pin != EXPECTED_PIN:
    raise SystemExit("Pin mismatch: reject the connection")

print("Pin matched: safe to continue")

This is the heart of public-key pinning: do not ask "is the chain trusted?" only. Also ask "does the presented public key match the one I pinned in advance?"

Pinning and TLS inspection are fundamentally incompatible. Mutual TLS (mTLS)

Standard TLS is one-sided: the server proves its identity to the client, but the client is anonymous at the TLS layer (authentication happens at the application layer via passwords, tokens, etc.). Mutual TLS flips this — both sides present certificates and prove possession of their private keys.

The mTLS Handshake Additions

In TLS 1.3, mTLS adds two extra messages to the handshake:

mTLS Additions to TLS 1.3 Handshake
Standard TLS 1.3 handshake proceeds as normal...

Server sends:  {EncryptedExtensions}
               {CertificateRequest}    ← NEW: server requests client cert
               {Certificate}            (server's cert)
               {CertificateVerify}      (server's proof)
               {Finished}

Client sends:  {Certificate}            ← NEW: client's X.509 cert chain
               {CertificateVerify}      ← NEW: signature proving client key ownership
               {Finished}

Both sides now authenticated at the TLS layer.

Where mTLS Is Used in Industry

Service-to-Service (Service Mesh)

This is the dominant use case today. In a microservices architecture, every service has its own identity certificate. Service meshes like Istio, Linkerd, and Consul Connect automate mTLS between all services — each sidecar proxy handles certificate issuance, rotation, and mutual verification. The application code never touches crypto.

SPIFFE/SPIRE: The SPIFFE standard (Secure Production Identity Framework For Everyone) defines a universal identity format for services: a URI like spiffe://cluster.local/ns/payment/sa/processor. SPIRE is the reference implementation — it issues short-lived X.509 certificates (SVIDs) to workloads, handles automatic rotation, and integrates with Kubernetes, AWS, and others. This is becoming the standard for service identity in cloud-native infrastructure.

See also: the Kubernetes Security guide’s deployment models and network policy sections for how these meshes sit inside different isolation patterns.

Zero Trust Architectures

BeyondCorp (Google's zero trust model) and similar architectures use client certificates to authenticate devices. Every corporate device gets a certificate from an internal CA. Network access decisions are based on device identity (from the cert) + user identity + device health, rather than "are you on the VPN?"

API Authentication

Financial services (PSD2/Open Banking), healthcare (FHIR APIs), and government systems commonly require mTLS for API access. The client cert replaces or supplements API keys, providing stronger cryptographic client identity and making accidental leakage via logs, screenshots, or copy/paste much less likely than bearer secrets.

Each device gets a unique client certificate provisioned at manufacturing time. When the device connects to the cloud backend, mTLS proves the device's identity. AWS IoT Core and Azure IoT Hub support this pattern. Google Cloud IoT Core used to, but Google retired it on August 16, 2023; the pattern still shows up in custom MQTT brokers and other managed IoT platforms.

mTLS with nginx — Configuration Example 
server {
    listen 443 ssl;
    server_name api.example.com;

    # Server certificate (standard TLS)
    ssl_certificate     /etc/nginx/certs/server.crt;
    ssl_certificate_key /etc/nginx/certs/server.key;

    # mTLS: require and verify client certificates
    ssl_client_certificate /etc/nginx/certs/client-ca.crt;  # CA that signed client certs
    ssl_verify_client on;           # "on" = required, "optional" = requested but not required
    ssl_verify_depth 2;             # max chain depth for client certs

    # Pass client cert info to upstream application
    location / {
        proxy_set_header X-Client-Cert-DN    $ssl_client_s_dn;
        proxy_set_header X-Client-Cert-Serial $ssl_client_serial;
        proxy_set_header X-Client-Cert-Verify $ssl_client_verify;
        proxy_pass http://backend;
    }
}

The upstream application receives the client's distinguished name and can make authorization decisions based on it — without ever handling TLS itself.

Certificate Lifecycle in a Service Mesh (Istio) 
1. Workload starts → Envoy sidecar generates a private key + CSR
2. CSR sent to istiod (control plane) via secure gRPC
3. istiod verifies workload identity (Kubernetes service account)
4. istiod signs the CSR with the mesh CA → returns short-lived cert (24h default)
5. Envoy uses the cert for all outbound/inbound mTLS connections
6. Before expiry, Envoy automatically requests a new cert (step 1-4)
7. No human intervention, no long-lived secrets, no manual rotation

Certificate chain:
  Mesh Root CA (self-signed or plugged into external CA)
    → istiod intermediate
      → Workload SVID (spiffe://cluster/ns/default/sa/my-service)

# --cacert          => trust only the CA that signed the server certificate
# If the server requires client auth and our cert is valid, the request succeeds.
curl \
  --cert client.crt \
  --key client.key \
  --cacert server-ca.crt \
  "${URL}"

# mtls_client.py
# Standard-library client example: prove possession of a client cert while
# also validating the server's certificate.

import ssl
import urllib.request

URL = "https://mtls.example.com/health"

# Trust the CA that issued the server certificate.
ctx = ssl.create_default_context(cafile="server-ca.crt")

# Present our own client certificate and private key for mTLS.
ctx.load_cert_chain(certfile="client.crt", keyfile="client.key")

with urllib.request.urlopen(URL, context=ctx) as response:
    print("HTTP status:", response.status)
    print(response.read().decode("utf-8"))

Unlike ordinary TLS, the client now proves identity during the handshake itself. If the client cert is missing, expired, or signed by the wrong CA, the handshake fails before the application sees an authenticated request.

DANE & DNSSEC

DNS-Based Authentication of Named Entities (DANE, RFC 6698) is an alternative trust model that ties certificate verification to the DNS infrastructure instead of the CA system. It publishes certificate information in TLSA DNS records, secured by DNSSEC.

The Problem DANE Solves

The CA trust model has a fundamental weakness: any of the ~100+ root CAs trusted by your browser can issue a certificate for any domain. DigiCert can issue a cert for google.com just as easily as Google's own CA can. DANE says: "The domain owner should decide which certificates are valid for their domain, and they express this via their DNS records."

How It Works

1. Domain owner publishes a TLSA record in DNS:

   _443._tcp.example.com. IN TLSA 3 1 1 (
     2bb183af2010f22323ac56... )    ← SHA-256 of the server's public key

2. The entire DNS chain is signed with DNSSEC:
   Root Zone (.) → .com → example.com → _443._tcp.example.com

3. Client resolves the TLSA record, validates DNSSEC signatures
   all the way up to the root, then compares the hash to the
   server's actual certificate/key during the TLS handshake.

TLSA Record Parameters

FieldValueMeaning
Usage0 (PKIX-TA)Pin a CA, still require PKIX validation
1 (PKIX-EE)Pin the leaf cert, still require PKIX validation
2 (DANE-TA)Pin a CA, DANE replaces PKIX (no CA trust store needed)
3 (DANE-EE)Pin the leaf, DANE replaces PKIX (most common)
Selector0 = Full cert, 1 = Public key only
Matching0 = Exact, 1 = SHA-256, 2 = SHA-512
DANE-EE (3 1 1) is the sweet spot: It pins the server's public key via its SHA-256 hash, and entirely bypasses the CA system. You can use a self-signed cert and it's perfectly valid as long as the TLSA record matches and the DNSSEC chain validates. This completely eliminates the need to trust third-party CAs.

DANE Adoption

DANE is widely deployed for email (SMTP with DANE via RFC 7672 — major providers like Comcast, mail.de, and many European ISPs validate DANE for mail delivery). For HTTPS in browsers, adoption is minimal — no major browser validates DANE natively because DNSSEC validation in stub resolvers is still unreliable and adds latency. However, for server-to-server communication (mail, XMPP, SIP) where you control the resolver, DANE is a powerful alternative to the CA model.

HSTS & Preloading

HTTP Strict Transport Security (HSTS, RFC 6797) defends against SSL stripping attacks — where an attacker downgrades an HTTPS connection to HTTP via a man-in-the-middle on the initial plaintext request.

The SSL Stripping Attack

Without HSTS:
  User types "example.com" in browser
    → Browser sends HTTP request to http://example.com
    → Attacker intercepts, proxies to real site over HTTPS
    → User sees http:// (no padlock) but most users don't notice
    → Attacker reads all traffic in plaintext

With HSTS:
  User types "example.com" in browser
    → Browser has cached HSTS policy
    → Browser INTERNALLY redirects to https://example.com
    → Never sends plaintext HTTP — attacker has nothing to intercept

The HSTS Header

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
                           ┃                  ┃                  ┃
                           ┃                  ┃                  └ Request inclusion
                           ┃                  ┃                    in browser preload list
                           ┃                  └ Apply to ALL subdomains
                           └ Remember for 1 year (seconds)

HSTS Preloading

HSTS has a first-visit problem: the very first time a user visits your site, they haven't received the HSTS header yet, so that first request is vulnerable. HSTS preloading solves this by hardcoding your domain into the browser itself. Chrome maintains the authoritative preload list (hstspreload.org), and Firefox, Safari, and Edge all consume it.

Once preloaded, your domain will never be accessed over plain HTTP by any browser — even on the very first visit, even if the user has never been to your site before. The downside: it's extremely difficult to remove a domain from the preload list, and it applies to all subdomains. If you have any subdomain that doesn't support HTTPS, preloading will break it.

Preloading is a one-way door. The removal process takes months (waiting for browser release cycles). Plan your subdomain infrastructure carefully before submitting. Every subdomain, including internal ones like staging.example.com or intranet.example.com, must support HTTPS.

CAA Records (Certification Authority Authorization)

CAA (RFC 8659) is a DNS record type that specifies which Certificate Authorities are authorized to issue certificates for your domain. CAs are required to check CAA records before issuance — it's enforced by the CA/Browser Forum Baseline Requirements.

# Only Let's Encrypt and DigiCert can issue for example.com:
example.com. IN CAA 0 issue "letsencrypt.org"
example.com. IN CAA 0 issue "digicert.com"

# No CA can issue wildcard certs:
example.com. IN CAA 0 issuewild ";"

# Alert security team if a violation is attempted:
example.com. IN CAA 0 iodef "mailto:security@example.com"

CAA is a lightweight defense-in-depth measure. It doesn't replace CT or pinning — a compromised CA could simply ignore CAA checks. But for honest CAs (which is most of them), it prevents accidental mis-issuance and limits your attack surface. Combined with CT monitoring, it gives you both prevention (CAA) and detection (CT).

The defense stack for domain owners: CAA records (restrict which CAs can issue) + Certificate Transparency monitoring (detect any cert issued for your domain) + HSTS preloading (prevent downgrade attacks) + short-lived certs (reduce exposure window). This layered approach doesn't depend on any single mechanism being perfect.
Lab Example — Bash: Query TLSA, HSTS, and CAA Records 
HOST=example.com

# DANE / TLSA:
# Query the TLSA record a DANE-aware client would compare against the server cert.
dig +dnssec "_443._tcp.${HOST}" TLSA

# HSTS:
# Fetch only headers and look for the Strict-Transport-Security policy.
curl -I "https://${HOST}" | grep -i '^strict-transport-security:'

# CAA:
# Ask DNS which certificate authorities are allowed to issue for the zone.
dig "${HOST}" CAA +short

This is a nice operational triad: DNS tells you about CA restrictions and DANE policy, while the HTTPS response tells you whether HSTS is actually being delivered to browsers.

Lab Example — Python: Verify an HSTS Header Is Present 
# check_hsts.py
# Quick operational check: confirm that the site is actually returning
# the Strict-Transport-Security header over HTTPS.

import ssl
import urllib.request

URL = "https://example.com/"

This does not check preload-list status, but it does verify the prerequisite: the site is actively serving an HSTS policy over a validated HTTPS connection.

Signed HTTP Exchanges (SXG) & Web Packaging

Signed Exchanges let a publisher cryptographically sign an HTTP response so it can be served by a third party (like a CDN or cache) while the browser still shows the original publisher's URL and trust indicators. This decouples content delivery from content origin.

How SXG Works

1. Publisher (example.com) generates a signed exchange:
   - The HTTP response (headers + body) is signed with the publisher's
     private key using a special "CanSignHttpExchanges" certificate
   - The signature includes the request URL, response headers, and body

2. A distributor (Google AMP cache, CDN) can now serve this content
   - The browser receives the signed bundle
   - Verifies the signature against the publisher's certificate
   - Displays "example.com" in the address bar, NOT the CDN's URL
   - The user sees the original origin with full trust indicators

The primary use case was Google's AMP project — allowing Google to serve cached pages from its own servers while showing the publisher's URL. SXG is a Chrome-only feature with limited adoption, but the underlying concept of signed content that's verifiable regardless of delivery path is architecturally interesting.

DPoP & Token Binding

Standard bearer tokens (like OAuth access tokens) have a fundamental problem: if someone steals the token, they can use it from anywhere. There's no cryptographic binding between the token and the legitimate client. Two mechanisms attempt to fix this:

DPoP (Demonstrating Proof of Possession)

DPoP (RFC 9449) binds OAuth tokens to a client-held asymmetric key pair. The client creates or provisions a DPoP key pair and includes a signed proof JWT with the token request. The authorization server binds the issued access token to that public key, typically via a JWK thumbprint confirmation (cnf.jkt). On every protected API call, the client must include a fresh DPoP proof signed with the same key.

DPoP Token Binding Flow
🔑
1. Client generates a proof key pair
This key pair belongs to the OAuth client instance. The private key stays on the client and is used only to sign DPoP proof JWTs.
📝
2. Token request carries a signed DPoP proof
The proof JWT covers the HTTP method and token endpoint URI. The authorization server verifies the signature and binds the new token to the client's public key.
📄
3. Access token contains proof-of-possession binding
If the token is a JWT, the binding is often represented in cnf.jkt. If the token is opaque, the resource server can learn the binding via introspection.
🛡
4. Every API call includes a fresh proof
The client sends both the token and a new DPoP proof containing htm, htu, iat, and for resource access, ath. Replay gets much harder because the proof is request-specific and time-bound.
Without DPoP:
  Token stolen → Attacker uses it freely from any device

With DPoP:
  Token stolen → Attacker also needs the private key
                 (which never left the client device)
                 → Token is useless without the key

Flow:
  1. Client generates proof key pair (kc_pub, kc_priv)
  2. Client → Auth Server: token request + DPoP Proof (JWT signed by kc_priv)
  3. Auth Server → Client: access_token (bound to kc_pub via "cnf.jkt" or equivalent token metadata)
  4. Client → API: access_token + fresh DPoP Proof (signed by kc_priv)
  5. API verifies: DPoP proof signature matches the key in the token's "cnf" claim

Token Binding (Deprecated but Conceptually Important)

Token Binding was an earlier attempt that operated at the TLS layer — it bound tokens to the TLS connection's key material. The client generated a key pair tied to the TLS session, and tokens were bound to that key. It was more comprehensive than DPoP but required TLS extension support from both sides, which proved too difficult to deploy. Chrome removed support in 2020. DPoP is the spiritual successor that works at the application layer instead.

The principle: Both DPoP and Token Binding apply the same crypto concept as mTLS — prove you possess a private key — but at the token/application layer rather than the transport layer. This pattern of "proof of possession" is fundamental to modern authentication: don't just present a credential, prove you own the key behind it.

Modern Encryption Protocols Beyond TLS

TLS dominates transport security, but it's not the only game. As a crypto engineer, you should know the broader landscape of modern protocols that solve different problems.

age — Simple File Encryption

age (Actually Good Encryption) by Filippo Valsorda is a modern replacement for PGP for file encryption. It's intentionally minimal: X25519 for key exchange, ChaCha20-Poly1305 for encryption, HKDF for key derivation. No config, no algorithm negotiation, no legacy baggage. The spec fits on one page.

# Encrypt to a recipient's public key:
age -r age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p \
    -o secret.txt.age secret.txt

# Decrypt with your private key:
age -d -i key.txt -o secret.txt secret.txt.age

# That's it. No key servers, no trust models, no algorithm choices.
Why age matters: PGP/GPG has been widely criticized by cryptographers (the "PGP Problem" by Latacora is worth reading). It has a massive attack surface: multiple algorithms, complex packet formats, key management disasters, and usability problems. age demonstrates that you can build a secure, usable encryption tool with modern primitives and no flexibility — flexibility is the enemy of security in this context.

MLS — Messaging Layer Security

MLS (RFC 9420) is the IETF standard for end-to-end encrypted group messaging. It solves a problem Signal's Double Ratchet doesn't: efficiently encrypting messages for large groups. With Signal's protocol, sending to a group of N people requires N separate encryptions. MLS uses a tree-based ratchet (TreeKEM) so the cost scales logarithmically.

Signal Double Ratchet (pairwise):
  Group of 1000 → sender encrypts 1000 times → O(N)

MLS TreeKEM:
  Group of 1000 → sender updates tree path → O(log N) ≈ 10 operations

MLS is being adopted by major platforms. Cisco Webex and Wire use it, and it's the encryption layer behind IETF's More Instant Messaging Interoperability (MIMI) effort which aims to enable interoperable E2EE messaging across platforms (driven partly by the EU's Digital Markets Act requirements).

NTS — Network Time Security

NTS (RFC 8915) adds cryptographic authentication to the Network Time Protocol (NTP). This matters because accurate time is a dependency for TLS certificate validation (checking notBefore/notAfter), OCSP response freshness, TOTP tokens, Kerberos tickets, and log correlation. If an attacker can shift your clock via NTP spoofing, they can make expired/revoked certificates appear valid or replay expired authentication tokens.

NTS uses a TLS handshake for the initial key exchange (NTS-KE), then uses the derived keys to authenticate subsequent NTP packets with AEAD (AES-SIV). This means NTP time synchronization gets integrity protection without the overhead of TLS on every packet.

WireGuard — Minimal VPN Protocol

WireGuard is worth studying as a masterclass in protocol minimalism. The entire codebase is ~4,000 lines of kernel code (compare to OpenVPN's ~100,000+). It uses a custom handshake inspired by the Noise IK pattern, with DoS-mitigation extensions (including cookie-based load limiting not present in standard Noise), ChaCha20-Poly1305 for transport, BLAKE2s for hashing, Curve25519 for DH, and HKDF for key derivation. There's no algorithm negotiation — if any primitive is broken, the protocol version changes. This "cryptographic versioning" approach eliminates downgrade attacks by design.

Message Authentication Codes (MACs)

A MAC answers one question: "Was this message tampered with?" It's a tag computed from a message + a secret key. Only someone with the same key can produce or verify the tag. Unlike a digital signature, MACs are symmetric — both sides share the same secret key.

How a MAC Works
📝
Sender has a message
"Transfer $500 to Alice"
🔑
Sender computes MAC
MAC(key, message) → tag: a3f8b2...
📨
Sends message + tag together
"Transfer $500 to Alice" + a3f8b2...
✅
Receiver verifies
Recomputes MAC(key, message) with the same key.
If the tags match → message is authentic and untampered.

If an attacker changes the message to "Transfer $5000 to Eve", the tag won't match when the receiver recomputes it. Without the key, the attacker can't forge a valid tag.

The Three MACs You'll See Everywhere

HMAC

Built on any hash function (SHA-256, SHA-384). The workhorse of TLS, JWT, API authentication, and basically everything.

HMAC(key, msg) =
  H((key ⊕ opad) ‖
    H((key ⊕ ipad) ‖ msg))

Two nested hashes.
Provably secure if the
hash function is secure.
Poly1305

A one-time authenticator. Extremely fast, used inside ChaCha20-Poly1305 (the AEAD cipher in TLS 1.3, WireGuard, etc.).

Poly1305(r, s, msg) =
  ((msg as polynomial
    evaluated at point r)
    mod 2¹³⁰-5) + s

One-time key per message.
Never reuse (r, s) pair!
CMAC

Built on a block cipher (AES). Common in banking, payment cards (EMV), and hardware security modules.

CMAC(key, msg) =
  AES-CBC on message blocks
  with special handling
  of the last block.

Used when you already
have AES hardware.

Digital Signatures — The Foundation of Everything Else

Digital signatures underpin code signing, package verification, GPG, and certificates themselves. Here's the simple mental model:

How Digital Signatures Work
📄
1. Hash the data
SHA-256(file) → digest: 7f83b1...
Reduces any size data to a fixed-size fingerprint.
🔒
2. Sign the hash with private key
Sign(private_key, digest) → signature: 3045...
Only the private key holder can produce this.
📦
3. Distribute file + signature + public key
The file, the signature, and the signer's public key are all public.
🔍
4. Anyone can verify
Verify(public_key, digest, signature) → true/false
Recompute hash of file, check signature against public key.

Linux Code Signing

Linux doesn't have a single unified code signing system, but individual pieces are signed:

Secure Boot: UEFI firmware verifies the bootloader's signature (Microsoft's UEFI CA or your own enrolled key) → bootloader verifies the kernel signature → kernel can verify module signatures. This is a chain of trust from firmware to running code.

Kernel modules: CONFIG_MODULE_SIG requires kernel modules to be signed with a key embedded at build time. Loading an unsigned module fails or triggers a taint warning.

IMA/EVM (Integrity Measurement Architecture): Extended attributes on files store signatures or hashes. The kernel verifies them before execution. Used in high-security deployments (government, financial).

GPG / PGP

PGP (Pretty Good Privacy) was created in 1991. GPG (GNU Privacy Guard) is the free, open-source implementation of the OpenPGP standard (RFC 4880). It's used for email encryption, file signing, and package signing (apt, RPM). It works, but it's widely criticized for usability and design issues.

The GPG Trust Model: Web of Trust

Unlike PKI (which has a hierarchy of CAs), PGP uses a decentralized "Web of Trust" — users sign each other's keys to vouch for identity.

PKI (Hierarchical) vs. PGP (Web of Trust)
PKI — Certificate Authority Model
        Root CA
       /      \
  Intermediate  Intermediate
    /    \          |
  Leaf  Leaf      Leaf

One authority decides trust.
Top-down. Centralized.
Simple for users.
PGP — Web of Trust
  Alice ←→ Bob
    ↕         ↕
  Carol ←→ Dave
    ↕
   Eve

Users sign each other's keys.
Peer-to-peer. Decentralized.
Complex for users.

How GPG Signing Works

GPG Sign & Verify Flow
🔑
1. Generate a key pair
gpg --full-generate-key
Creates a master key (for signing/certification) + subkeys (for encryption, signing).
Your public key has a fingerprint like 3AA5 C343 F9BB...
📋
2. Publish your public key
Upload to a keyserver (keys.openpgp.org) or share directly.
Others can find your key by email or fingerprint.
📝
3. Sign a file
gpg --detach-sign --armor release.tar.gz
Produces release.tar.gz.asc (ASCII-armored detached signature).
Internally: SHA-256 hash of file → signed with your private key.
✅
4. Anyone verifies
gpg --verify release.tar.gz.asc release.tar.gz
GPG fetches your public key → verifies signature → confirms file is untampered and came from you.
GPG for Email Encryption (How It Actually Works) 
Sending an encrypted + signed email to Bob:

1. Alice composes the email plaintext

2. Alice SIGNS the email:
   SHA-256(plaintext) → digest
   Sign(alice_private_key, digest) → signature
   Attach signature to email

3. Alice ENCRYPTS the email:
   Generate random session key (e.g., AES-256 key)
   AES-256(session_key, plaintext + signature) → ciphertext
   RSA-Encrypt(bob_public_key, session_key) → encrypted_session_key
   Send: encrypted_session_key + ciphertext

4. Bob DECRYPTS:
   RSA-Decrypt(bob_private_key, encrypted_session_key) → session_key
   AES-256-Decrypt(session_key, ciphertext) → plaintext + signature

5. Bob VERIFIES:
   SHA-256(plaintext) → digest
   Verify(alice_public_key, digest, signature) → valid!

This is "hybrid encryption" — asymmetric crypto wraps a symmetric key,
symmetric crypto does the heavy lifting on the actual data.
Same pattern used in TLS, age, and almost every encryption system.

The Problems with GPG

Why cryptographers criticize GPG:

Key management nightmare: Users must manually manage keys, keyrings, trust levels, subkeys, revocation certificates, keyserver uploads, and key signing parties. Most people get this wrong.

Algorithm soup: GPG supports dozens of algorithms for backward compatibility (3DES, IDEA, CAST5, ElGamal, DSA, etc.). This makes implementation complex and the attack surface huge.

No forward secrecy: If your long-term key is compromised, all past encrypted messages can be decrypted. Compare this to Signal's Double Ratchet where keys rotate per-message.

Metadata leaks: PGP encrypts the body but not the subject line or headers. An observer knows who is talking to whom and when.

Usability: Even after 30+ years, no one has built a truly usable PGP email client. The EFAIL vulnerability (2018) showed that most email clients implemented PGP decryption insecurely.

For file signing specifically, GPG remains common (it's what apt, git commits, and most open-source releases use). For encrypted communication, Signal/MLS/age have largely superseded it. For package signing, Sigstore is the modern replacement.

Sigstore — The Future of Software Signing

Sigstore is an open-source project (backed by the Linux Foundation) that solves the biggest problem with traditional code/package signing: key management. With GPG, you need to generate keys, protect them, distribute public keys, and deal with revocation. Most developers skip signing entirely because it's too hard. Sigstore makes signing as easy as logging in.

How Keyless Signing Works

Sigstore Keyless Signing Flow
👤
1. Developer authenticates via OIDC
Log in with Google, GitHub, or Microsoft identity.
"I am developer@gmail.com" — proven via OAuth/OIDC token.
🔑
2. Fulcio (CA) issues an ephemeral certificate
Sigstore's CA verifies the OIDC token → issues a short-lived certificate (10 minutes) binding the developer's identity to an ephemeral key pair. No long-term keys to manage!
📝
3. Developer signs the artifact
Sign the package/container/binary with the ephemeral private key. The signature + certificate are produced.
📗
4. Rekor (transparency log) records the signing event
The signature is logged in an append-only transparency log (like CT for certificates). This proves the signing happened while the ephemeral cert was valid. The ephemeral private key is then discarded.
✅
5. Verifier checks everything
Verify signature is valid → cert was issued by Fulcio → signing was logged in Rekor while cert was valid → identity matches expected signer. No public keys to fetch or trust!

Sigstore Components

ComponentRoleAnalogy
FulcioCertificate Authority — issues ephemeral certs based on OIDC identityLike Let's Encrypt, but for code signing and identity-based
RekorTransparency log — immutable record of all signing eventsLike Certificate Transparency logs
CosignCLI tool for signing/verifying container imagesLike gpg --sign but for containers
Sigstore-python / sigstore-jsLanguage clients for signing packagesLike GPG libraries but using Sigstore
Traditional (GPG) Signing

Generate key pair → protect private key forever → publish public key to keyserver → sign artifacts → if key compromised, revoke and re-sign everything → users must fetch and trust your key

Result: most developers don't bother.

Sigstore (Keyless) Signing

Log in with GitHub/Google → get ephemeral cert → sign → cert expires in 10 min → transparency log proves it happened → verifiers check identity, not keys

Result: signing is as easy as cosign sign.

The paradigm shift: Traditional signing asks "do you trust this key?" — which requires you to somehow obtain and verify the right public key. Sigstore asks "do you trust this identity?" — which maps to something humans already understand (email addresses, GitHub accounts). The transparency log replaces the need for long-term key management by proving that signing happened at a specific time by a specific identity.

Where Sigstore Is Used Today

npm (registry signatures), Python/PyPI (PEP 761), Kubernetes (all release artifacts), Homebrew (bottle signatures), and GitHub Actions (artifact attestations via actions/attest). It's becoming the default for open-source supply chain security. If you're building anything in the software supply chain space, Sigstore is essential knowledge.

Software Supply Chain Threat Model

Before locking things down, you need to understand where the attacks happen. The software supply chain is every step between a developer writing code and that code running in production. Each step is an attack surface.

The Software Supply Chain — Attack Points
⚠️
1. Developer Workstation
Attacks: Compromised dev machine, stolen credentials, malicious IDE extensions, social engineering.
Examples: Codecov breach (2021) — attacker modified the Bash uploader script in CI to exfiltrate env vars.
⚠️
2. Source Code Repository
Attacks: Unauthorized commits, force pushes to main, compromised CI config, malicious PRs that bypass review.
Examples: PHP self-hosted git server compromise (2021) — attacker pushed backdoored commits directly.
⚠️
3. Dependencies
Attacks: Typosquatting, dependency confusion, compromised maintainer accounts, malicious updates.
Examples: event-stream (2018) — maintainer handed off npm package to attacker who added crypto-stealing code. ua-parser-js (2021) — hijacked npm account pushed cryptominer.
⚠️
4. Build System / CI-CD
Attacks: Tampered build scripts, compromised CI runners, injected steps, poisoned build cache.
Examples: SolarWinds (2020) — attacker modified the build process to inject a backdoor into the compiled binary. The source code was clean; the build was not.
⚠️
5. Package Registry / Artifact Store
Attacks: Compromised registry, man-in-the-middle on downloads, malicious mirrors.
Examples: PyPI malware campaigns — thousands of typosquatted packages uploaded each year.
⚠️
6. Deployment
Attacks: Tampered artifacts in transit, compromised deployment credentials, unauthorized deployments.
Examples: Container image tag mutation — latest can be overwritten with a malicious image at any time.
The SolarWinds lesson: The most sophisticated supply chain attacks don't touch the source code at all. SolarWinds' source repo was clean — the attacker injected malware during the build process. This means code review alone is insufficient. You need to verify the entire path from source to deployed artifact, which is exactly what SLSA addresses.

SLSA Framework (Supply-chain Levels for Software Artifacts)

SLSA (pronounced "salsa") is a security framework from Google that defines increasing levels of supply chain integrity. Think of it as a maturity model — each level adds more protections. It answers: "How confident are you that this artifact was built from the source code you think it was?"

SLSA Levels — Progressive Supply Chain Security
L0
No guarantees
No provenance. You have no idea how this artifact was built or from what source. This is where most software starts.
L1
Provenance exists
The build process generates provenance metadata — a signed statement saying "artifact X was built from source Y using build process Z." The provenance exists but isn't tamper-proof yet.
L2
Hosted build + signed provenance
Build runs on a hosted service (GitHub Actions, Cloud Build). Provenance is signed by the build service, not the developer. This prevents a compromised dev machine from forging provenance.
L3
Hardened builds
Build runs in an isolated, ephemeral environment. Build service is resistant to insider threats. Provenance is unforgeable — even the build service admins can't fake it. Prevents SolarWinds-style attacks.

What Provenance Looks Like

{
  "_type": "https://in-toto.io/Statement/v1",

    "name": "my-app",

# "Artifact with hash abc123 was built from org/repo at tag v1.2.3
#  by GitHub Actions runner, run ID 123."
#
# A verifier checks: is the signature valid? Does the artifact hash match?
# Is the source repo the one I expect? Was the builder trusted?

# .github/workflows/release.yml
name: Release with SLSA Provenance

on:
  push:
    tags: ['v*']

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      digest: ${{ steps.hash.outputs.digest }}
    steps:
      - uses: actions/checkout@v4
      - run: make build
      - name: Compute digest
        id: hash
        run: echo "digest=$(sha256sum my-app | cut -d' ' -f1)" >> "$GITHUB_OUTPUT"
      - uses: actions/upload-artifact@v4
        with:
          name: my-app
          path: my-app

    needs: build
    permissions:
      actions: read
      id-token: write       # Needed for Sigstore signing
      contents: write
    uses: slsa-framework/slsa-github-generator/.github/workflows/generator_generic_slsa3.yml@v2.1.0
    with:
      base64-subjects: "${{ needs.build.outputs.digest }} my-app"

  for commit in $(git rev-list $oldrev..$newrev); do

  done
done

Git & Version Control Hardening

Git itself has several security features and risks that organizations need to manage. Here's a comprehensive hardening guide.

Git's Built-in Security Properties

Git's Content-Addressable Storage = Built-in Integrity
📈
Every object is content-addressed
Every blob, tree, commit, and tag is identified by its SHA-1 hash (SHA-256 in newer versions). Change one byte anywhere in history, and every commit hash from that point forward changes.
🔗
Commits form a hash chain
Each commit hash includes the hash of its parent commit(s) and the hash of the tree (all files). This is structurally similar to a blockchain — tampering with any historical commit breaks the chain.
🔒
Signed tags anchor the chain
A signed tag on a commit cryptographically binds a human-verified identity to a specific point in the hash chain. If you trust the tag signature and verify the hash chain backward, you can trust the entire history.
# The hash chain in practice:
commit abc123
  parent: def456        ← hash of parent commit
  tree: 789fab          ← hash of ALL files at this point
  author: dev@co.com
  message: "Add feature"

# abc123 = SHA-1(parent_hash + tree_hash + author + timestamp + message)
# Change ANYTHING → hash changes → all child commits become invalid
SHA-1 collision concern: Git historically uses SHA-1, which has known collision attacks (SHAttered, 2017). Git mitigated this with a hardened SHA-1 implementation (detects the known collision patterns) and is transitioning to SHA-256 (git init --object-format=sha256). For most threat models the hardened SHA-1 is sufficient, but for high-security environments, plan for the SHA-256 migration.

Secrets Management

Secrets (API keys, database credentials, signing keys, tokens) are the most common thing that goes wrong in supply chain security. The rules are simple but widely ignored.

The Golden Rules

Where Secrets Should (and Should Not) Live
Never Put Secrets Here

Source code / git history — once committed, it's there forever (even after deletion)

CI/CD config files — .github/workflows/ files are public in public repos

Container images — layers are inspectable with docker history

Environment variables in Dockerfiles — ENV SECRET=... is baked into the image

Log output — secrets printed to stdout end up in log aggregators

Secrets Belong Here

Dedicated vault — HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager, Azure Key Vault

CI/CD secret stores — GitHub Actions Secrets, GitLab CI Variables (masked + protected)

Runtime injection — mounted as files or env vars at deploy time, never baked into artifacts

Short-lived credentials — OIDC federation (GitHub → AWS via AssumeRoleWithWebIdentity) instead of long-lived keys

The Modern Approach: No Long-Lived Secrets

OIDC Federation — Eliminating Static Credentials in CI
💻
1. GitHub Actions workflow starts
The workflow requests an OIDC token from GitHub's identity provider. This token says "I am repo org/app, branch main, workflow deploy.yml."
🔑
2. AWS verifies the OIDC token
AWS IAM is pre-configured to trust GitHub's OIDC provider. It validates the token's signature and claims (repo, branch, workflow).
🔔
3. AWS issues temporary credentials
AssumeRoleWithWebIdentity returns short-lived AWS credentials (15 min to 1 hour). No static AWS keys stored anywhere.
✅
4. Credentials expire automatically
After the workflow finishes, the credentials are useless. Nothing to leak, nothing to rotate. Even if logs are exposed, the creds are already expired.

CI/CD Pipeline Security

Your CI/CD pipeline is the most valuable target in your infrastructure. It has access to your source code, your secrets, your signing keys, and your production environment. Compromising CI gives an attacker everything.

The Threat Model for CI/CD

CI/CD Attack Surfaces
⚠️
Workflow file injection
Attacker submits a PR that modifies .github/workflows/ to exfiltrate secrets. If workflows run on PRs from forks without restrictions, secrets can leak.
Fix: Use pull_request trigger (not pull_request_target) for untrusted code. Require approval for fork PRs.
⚠️
Dependency poisoning in CI
Build step runs npm install and a compromised dependency executes a postinstall script that exfiltrates $GITHUB_TOKEN or other env vars.
Fix: Pin dependencies with lockfiles + hash verification. Use --ignore-scripts where possible. Network-restrict CI runners.
⚠️
Compromised CI runner
Self-hosted runners that aren't ephemeral can be compromised by one build and affect subsequent builds. Credentials from previous builds may persist.
Fix: Use ephemeral runners (fresh VM per job). Never run untrusted code on persistent self-hosted runners.

Authentication Methods — The Landscape

Authentication answers one question: "Are you who you claim to be?" It is distinct from authorization (what you're allowed to do) and identification (who you are). Every authentication system is built on one or more of three factors, and every factor has a cryptographic mechanism backing it.

The Three Factors of Authentication
Something You Know

Password, PIN, security question, passphrase

Crypto backing: Password hashing (Argon2, bcrypt), KDF-derived keys

Weakness: Can be guessed, phished, or leaked from databases

Something You Have

TOTP app, hardware key (YubiKey), smart card, phone

Crypto backing: HMAC-SHA1 (TOTP/HOTP), ECDSA (WebAuthn), symmetric shared secret

Weakness: Can be lost, stolen, or SIM-swapped (for SMS)

Something You Are

Fingerprint, face ID, iris scan, voice

Crypto backing: Biometric template stored in secure enclave, compared locally, private key released on match

Weakness: Can't be changed if stolen. Local comparison trust depends on device security.

Passwords & Cryptographic Hashing

Passwords are the most widely deployed authentication method, and the most widely breached. The crypto challenge: a server must be able to verify a password without storing the password itself — because databases get breached.

Why You Can't Just Hash a Password

The Problem with SHA-256(password)
⚠️
Problem 1: Precomputed rainbow tables
An attacker precomputes SHA-256 of every common password offline. When they steal your database, they just look up the hash. SHA-256("password123") is always the same value.
⚠️
Problem 2: Speed is the enemy
SHA-256 is designed to be fast — modern GPUs can compute billions of SHA-256 hashes per second. This means brute-force attacks are feasible. We need hashing that is intentionally slow.

The Solution: Password Hashing Functions

How Password Hashing Works
🎲
1. Generate a random salt
salt = CSPRNG(16 bytes) — unique per user, per password change.
Stored alongside the hash. Makes rainbow tables useless — attacker must crack each hash individually.
🕐
2. Hash with intentional cost
hash = Argon2id(password, salt, time=3, memory=64MB, parallelism=4)
Takes ~300ms on a server. Takes 300ms × millions of attempts on an attacker's GPU cluster = impractical.
💾
3. Store the output string
$argon2id$v=19$m=65536,t=3,p=4$c2FsdGhlcmU$hash...
The algorithm, version, parameters, salt, and hash are all encoded together. No separate salt column needed.
✅
4. Verify by re-hashing
Parse the stored string → extract salt + params → re-run Argon2id(input_password, salt, same_params) → compare in constant time.

After the full database leaked, crackers cracked ~90% of passwords within days using GPU clusters and rainbow tables. This is why password hashing functions exist.

Session Tokens & Cookies

After a user authenticates (proves their identity once), the server issues a token so they don't have to re-authenticate on every request. This is the most common pattern in web applications.

Server-Side Session Token Flow
👤
1. User logs in with credentials
Server verifies password against stored Argon2 hash. Authentication succeeds.
🎲
2. Server generates a session token
token = CSPRNG(32 bytes) → base64url encode
Cryptographically random. Unpredictable. Stored server-side (Redis, DB) mapped to user ID + expiry.
🍪
3. Token sent to browser as cookie
Set-Cookie: session=TOKEN; HttpOnly; Secure; SameSite=Strict; Path=/
HttpOnly = JS can't read it (XSS protection). Secure = HTTPS only. SameSite=Strict = CSRF protection.
✅
4. Every request includes the cookie
Browser sends the cookie automatically. Server looks up the token in its session store → finds the associated user → authorizes the request.
The cryptographic property: The session token's security comes from its unpredictability — 32 bytes from a CSPRNG gives 2²⁵⁶ possible values. An attacker guessing tokens would need astronomically many attempts. There is no public-key or symmetric-key crypto in the token itself — the security is purely in the randomness.

JWT — JSON Web Tokens

JWTs are stateless tokens — the server doesn't need to store anything to verify them. The token itself is cryptographically signed and carries all the claims inside it.

JWT Structure
Header (base64url)
{"alg":"ES256","typ":"JWT"}
.
Payload (base64url)
{"sub":"user123","exp":1735689600,"iss":"auth.example.com"}
.
Signature
ECDSA(privKey, SHA256(header.payload))
JWT Flow — How It Works
👤
1. User authenticates
Auth server verifies credentials (password, social login, etc.)
📝
2. Auth server issues a signed JWT
Creates payload with claims (user ID, roles, expiry, issuer). Signs with its private key (ES256/RS256). Nothing is stored server-side.
📨
3. Client stores and sends JWT
Typically in memory or Authorization: Bearer TOKEN header. (Storing in localStorage risks XSS theft.)
✅
4. Resource server verifies without calling auth server
Verifies ECDSA signature using auth server's public key. Checks exp, iss, aud claims. No database lookup needed — stateless.

JWT Algorithm Pitfalls — Security Anti-Patterns

The "alg: none" attack: Some libraries trust the algorithm specified in the JWT header. An attacker can change "alg":"RS256" to "alg":"none" and strip the signature — the library accepts it as valid. Always hardcode the expected algorithm server-side, never read it from the token.
The HS256/RS256 confusion attack: If a server accepts both HMAC (symmetric) and RSA (asymmetric) algorithms, an attacker can forge a token signed with HS256 using the server's public RSA key as the HMAC secret (which is public!). Always specify one algorithm and reject anything else.

Public Key Authentication (SSH)

SSH key authentication is the clearest real-world example of asymmetric cryptography applied to authentication. The server stores your public key. You prove identity by signing a challenge with your private key — the secret never leaves your machine.

SSH Public Key Authentication — Step by Step
🔑
Setup (one time): Generate key pair
ssh-keygen -t ed25519 -C "dev@company.com"
Creates ~/.ssh/id_ed25519 (private — never leaves machine) and ~/.ssh/id_ed25519.pub (public — share freely).
📋
Setup: Server stores public key
ssh-copy-id user@server → appends public key to ~/.ssh/authorized_keys on server.
Server now knows your public key. Only your public key is stored — no secret on server.
💻
Login: Client announces key
Client says "I want to authenticate with Ed25519 key with fingerprint XYZ." Server checks if that public key is in authorized_keys.
🎲
Login: Server sends a random challenge
Server sends a challenge incorporating the current session ID. The client signs the challenge with its private key — the server then verifies the signature using the stored public key. No encryption of a nonce occurs; Ed25519 is a signature-only algorithm and cannot encrypt.
📝
Login: Client signs the challenge
signature = Ed25519_sign(private_key, challenge + session_id)
The session_id binds this to the specific connection — replay attacks are impossible.
✅
Login: Server verifies and grants access
Ed25519_verify(stored_public_key, challenge + session_id, signature)
If valid → authenticated. The private key never left the client machine at any point.
Why SSH key auth is stronger than passwords: A password can be phished (user types it into a fake site), guessed, or stolen from a breached DB. An SSH key can't be phished — the private key never leaves your machine, and the challenge-response is bound to the specific TLS session (no replay possible). The server never sees anything sensitive.

Multi-Factor Authentication & TOTP

MFA requires at least two factors — typically "something you know" (password) + "something you have" (phone/hardware key). Even if your password is stolen, the attacker also needs the second factor.

TOTP — Time-Based One-Time Password (RFC 6238)

TOTP is what your authenticator app (Google Authenticator, Authy) generates. It's based on a shared secret and the current time — both your phone and the server compute the same 6-digit code independently.

How TOTP Works Cryptographically
🔑
1. Setup: Share a secret seed
Server generates a random 160-bit secret key (the "TOTP seed"). Displayed as a QR code. Your authenticator app scans it and stores the secret. This seed is the shared secret — protect it like a password.
🕐
2. Every 30 seconds: Both sides compute the same code
T = floor(current_unix_time / 30) ← time step
HOTP = HMAC-SHA1(secret, T) ← HMAC of time
code = extract 6 digits from HOTP ← truncate
Your phone and the server run the exact same calculation.
✅
3. Server verifies your code
Computes TOTP for current time window ± 1 (to account for clock skew). If your code matches → authenticated. Code is useless after 30 seconds.

WebAuthn / Passkeys — Phishing-Proof Auth

WebAuthn (Web Authentication API, W3C standard) is the most cryptographically sound authentication method for the web. It uses public key cryptography where the credential is bound to the specific website's origin — making phishing attacks cryptographically impossible.

WebAuthn Registration & Authentication Flow
Registration (one time)
1
Server sends challenge
Random bytes + relying party ID (e.g., "example.com")
2
Device generates key pair
ECDSA P-256 or Ed25519. Private key stored in authenticator (secure enclave / hardware key). Bound to origin "example.com".
3
Public key sent to server
Server stores public key + credential ID. Private key never leaves the device.
Authentication (every login)
1
Server sends challenge
Random bytes. User must be on correct origin.
2
User verifies (biometric / PIN)
Authenticator requires user verification locally before releasing the key. Biometric template never leaves device.
3
Device signs challenge with origin binding
Sign(privKey, challenge + origin + rpId)
The origin "example.com" is cryptographically bound. Signing for "evil-example.com" produces a different, invalid signature.
Why phishing is cryptographically impossible with WebAuthn: The authenticator includes the current page's origin in the signed data. If an attacker tricks you into visiting evil-example.com, the signature will contain that origin, not example.com. The server will reject it. There is no way to use a WebAuthn credential on the wrong site — unlike TOTP codes or passwords which can be intercepted and replayed.

Passkeys = WebAuthn + Cloud Sync

A passkey is a WebAuthn credential that syncs across your devices via the platform's keychain (iCloud Keychain, Google Password Manager). Cryptographically identical to WebAuthn — the private key is synced in encrypted form using keys derived from your account credentials. This solves the "what if I lose my phone?" problem at the cost of trusting Apple/Google's sync infrastructure.

OAuth 2.0 & OpenID Connect (OIDC)

OAuth 2.0 is an authorization framework (not authentication — that confusion is a common source of security bugs). OIDC is an authentication layer built on top of OAuth 2.0. Together they're the mechanism behind "Sign in with Google/GitHub/Apple."

OAuth 2.0 Authorization Code Flow + PKCE
💻
1. App generates PKCE code verifier + challenge
code_verifier = CSPRNG(32 bytes) → base64url
code_challenge = base64url(SHA-256(code_verifier))
PKCE prevents authorization code interception — even if the code is stolen, the attacker doesn't have the verifier.
🔗
2. User redirected to Identity Provider with challenge
GET /authorize?client_id=...&code_challenge=HASH&state=RANDOM
The state parameter is a CSRF token — prevents cross-site request forgery on the redirect.
👤
3. User authenticates with IdP (Google, GitHub, etc.)
The IdP handles actual authentication (password + MFA). Your app never sees the user's password. IdP issues an authorization code.
📃
4. App exchanges code + verifier for tokens
POST /token { code, code_verifier, client_secret? }
IdP verifies: SHA-256(code_verifier) == code_challenge. Public clients (SPA/mobile) rely on PKCE and do not use a client secret. Confidential web apps also authenticate the client, for example with a client secret or private_key_jwt.
✅
5. App verifies the ID token (OIDC)
ID token is a JWT signed by the IdP (RS256 or ES256). App fetches IdP's public keys from JWKS endpoint → verifies signature → reads user identity from claims (sub, email).
OAuth Client Types — Same PKCE, Different Trust Boundaries
Public Client (SPA / Native App)

Runs on a user-controlled device or in the browser.

Uses Authorization Code + PKCE.

No durable client secret can be trusted here, so the token request is typically:
POST /token { code, code_verifier }

Confidential Client (Server Web App)

Runs on your backend and can protect credentials.

Often still uses Authorization Code + PKCE for defense in depth.

The backend also authenticates itself at the token endpoint:
POST /token { code, code_verifier, client_secret or private_key_jwt }

SAML & Enterprise SSO

SAML (Security Assertion Markup Language) is the older enterprise SSO standard, widely used in corporate environments with identity providers like Okta, Azure AD, and PingFederate. It predates OAuth and uses XML with XML Digital Signatures instead of JSON + JWT.

SAML SSO Flow (SP-Initiated)
💻
1. User accesses Service Provider (SP)
Tries to access app.company.com without a session.
🔗
2. SP redirects to Identity Provider (IdP)
Sends a signed SAMLRequest (XML) containing the SP's entity ID and a random RelayState (CSRF protection). The SAMLRequest may be signed with the SP's private key.
👤
3. User authenticates with IdP
Password + MFA with the company's identity provider (Okta, Azure AD, etc.)
📄
4. IdP sends signed SAMLResponse via browser
An XML document containing assertions (who the user is, their attributes, session expiry). The entire assertion is signed using XML DSig (RSA-SHA256 or ECDSA). Sent as an HTTP POST to the SP's Assertion Consumer Service URL.
✅
5. SP verifies the assertion
Validates XML signature against IdP's certificate (pre-configured). Checks IssueInstant, NotBefore, NotOnOrAfter, Audience. Creates a local session for the user.
SAML XML Signature Wrapping attacks: Because XML is complex, some SAML parsers verify a signature over one part of the document but read user identity from a different (unsigned) part. An attacker can wrap a valid signed assertion around a malicious one. Always use a well-tested SAML library and never implement XML signature verification yourself.

Password Authenticated Key Exchange: SRP & OPAQUE

Standard password authentication has a fundamental weakness: your password (or its hash) must be stored on the server. If the server is breached, attackers can crack the hashes offline. Password Authenticated Key Exchange (PAKE) protocols solve this — the server can verify your password without ever seeing it or storing it in a form that can be cracked.

SRP — Secure Remote Password (RFC 2945)

SRP is a Password Authenticated Key Exchange (PAKE) protocol based on Diffie-Hellman. The client proves it knows the password without sending it. The server stores a "verifier" derived from the password — useless for logging in even if stolen.

Setup (registration):
  client: x = H(salt || password)
  client: v = g^x mod N         ← the verifier (stored on server)
  client sends v to server — server stores (salt, v)

Authentication:
  Server: generates B = kv + g^b mod N (b = random)
  Client: generates A = g^a mod N (a = random)
  Both: exchange A and B
  Both: derive the same shared key S using DH math + password knowledge
  Both: prove they have the same S via HMAC exchange

  # The math ensures:
  # - Server can verify client knows the password (via S derivation)
  # - Client can verify server knows the verifier (mutual auth)
  # - Password never sent over the wire
  # - Verifier is useless without the actual password for login

OPAQUE — The Modern Standard

OPAQUE (Oblivious Pseudorandom Functions-based Authenticated Key Exchange, RFC 9807) is the modern standardized augmented PAKE that improves on SRP. It uses Oblivious PRFs — a primitive where the server helps the client derive key material from the password without learning the password, and without the client learning the server's secret.

OPAQUE Login Flow (High Level)
📝
1. Registration stores an encrypted credential envelope
Instead of storing a normal password hash for later offline guessing, the server stores OPAQUE-specific data: an OPRF-related record plus an envelope that the client can unlock only with the correct password.
🔓
2. Client blinds the password during login
The client sends a blinded OPRF input and an ephemeral key share. The server helps evaluate the OPRF without learning the password itself.
🔐
3. Server returns OPRF output + envelope + its key share
The client unblinds the OPRF result, derives the password-based secret, unlocks the stored envelope, and verifies the server's response.
✅
4. Both sides finish with mutual authentication
Client and server prove they derived the same session secret. The result is authenticated key exchange without ever sending the password and without leaving a conventional offline password oracle in the database.
OPAQUE vs. Traditional Password Auth
Traditional (Argon2 hash)

Server stores Argon2id(password, salt)

If DB is stolen → attacker can attempt to crack hashes offline at their leisure using GPU clusters.

Even with strong hashing, the hash exists and is a target.

OPAQUE

Server stores OPAQUE-specific credential material rather than a normal password hash.

If DB is stolen → attacker does not get a conventional password-verification oracle they can grind through offline; they are pushed toward online attacks that can be rate-limited and monitored.

Server never sees the password. Client never learns the server's secret.

Concrete Example — Why a Breach Looks Different from Argon2id 
Traditional password database:
  { user, salt, Argon2id(password, salt) }

Offline attack after breach:
  guess -> Argon2id(guess, salt) -> compare -> repeat millions of times

OPAQUE-style database:
  { user, credential envelope, OPRF-related server record, public metadata }

After breach:
  attacker does not receive a normal password hash they can iterate against
  in a simple offline loop. To test guesses, they are pushed toward live
  interaction with the server, where rate limits, detection, lockouts,
  and step-up controls apply.
Adoption: SRP is deployed in Apple iCloud Keychain escrow, 1Password's authentication, and some VPN protocols. OPAQUE was standardized as RFC 9807 in 2025, but deployment is still early compared with passwords, WebAuthn, or OAuth. Today it's mainly seen in research, prototype work, and selective implementation efforts rather than broad mainstream login systems.

Kerberos & Active Directory

Kerberos is the authentication backbone of every Windows enterprise network. If your organization uses Active Directory (AD), every time you log into Windows, access a file share, or authenticate to an internal service, Kerberos is doing the work. It was designed at MIT in the 1980s and is still the dominant protocol for enterprise authentication today.

The Analogy: A Theme Park Ticket System

Kerberos is like a Theme Park
Theme Park Entrance

You show your ID at the gate, pay once, and get a wristband (your day pass). You never show your ID again inside the park.

= Kerberos AS (Authentication Server)

Ride Ticket Booth

You show your wristband at a booth and get a ride-specific ticket for Space Mountain. You can only use it for that ride.

= Kerberos TGS (Ticket Granting Server)

The Ride Itself

You hand the ride-specific ticket to the operator. They let you on. No passwords, no re-authentication.

= The service you're accessing (file server, email, etc.)

How Kerberos Authentication Works — Step by Step

Full Kerberos Authentication Flow
👤
Step 1 — AS-REQ: "I am Alice, let me in"
Client → KDC (Authentication Server):
The username (principal name) is sent in plaintext in the AS-REQ body. Only the pre-authentication data — { timestamp: now } — is encrypted with Alice's key (derived from her password via PBKDF).
The password never leaves Alice's machine. Only a key derived from it does.
🎫
Step 2 — AS-REP: KDC issues a Ticket Granting Ticket (TGT)
KDC → Client:
TGT = encrypted with KDC's secret key (only KDC can read it) — contains Alice's identity + session key.
Session Key = encrypted with Alice's key (only Alice can decrypt it).
Analogy: you got your wristband. TGT is valid for ~10 hours.
🎰
Step 3 — TGS-REQ: "I want to access the file server"
Client → KDC (Ticket Granting Server):
{ TGT, Authenticator } where Authenticator = timestamp encrypted with the session key.
TGS verifies: decrypts TGT with its own key → gets session key → decrypts authenticator → confirms timestamp is fresh.
🎫
Step 4 — TGS-REP: KDC issues a Service Ticket
KDC → Client:
Service Ticket = encrypted with the file server's secret key (only file server can read it) — contains Alice's identity + service session key.
Service Session Key = encrypted with Alice's TGT session key.
Analogy: you got your ride ticket for Space Mountain.
📄
Step 5 — AP-REQ: Present ticket to the service
Client → File Server:
{ Service Ticket, Authenticator } where new Authenticator is encrypted with service session key.
File server decrypts the Service Ticket with its own secret → gets Alice's identity and session key → verifies authenticator.
✅
Step 6 — AP-REP: Mutual authentication (optional)
File Server → Client: { timestamp + 1 } encrypted with service session key.
Proves the file server also knows the session key — mutual authentication. Client knows it's talking to the real server.
You're now accessing the file share. No passwords sent anywhere.

The Cryptography Inside Kerberos

# Key Derivation (from password):
Alice's long-term key = DeriveKey(password, salt=username+realm)
Historically: MD4(UTF-16LE(password)) — the NT Hash
Modern AES Kerberos: PBKDF2(HMAC-SHA1, password, salt, 4096 iterations) → AES-128 or AES-256

# TGT structure (simplified):
TGT = AES-256-CTS-HMAC-SHA1-96 encrypt(
  with: krbtgt account's key (the KDC's master key),
  data: {
    client_name: "alice",
    realm: "CORP.EXAMPLE.COM",
    auth_time: now,
    end_time: now + 10h,
    session_key: random_AES_key,
    flags: RENEWABLE | FORWARDABLE
  }
)

# Authenticator (proves ticket freshness, prevents replay):
Authenticator = AES encrypt(
  with: session_key,
  data: { client_name, timestamp, sequence_number }
)
# Timestamp must be within ±5 minutes of KDC time
# This is why accurate NTP is a Kerberos security dependency!
The krbtgt account is the crown jewel. The KDC's master key is derived from the domain's krbtgt account password. If an attacker gets this key (e.g., via a DCSync attack), they can forge any Kerberos ticket for any user — including domain admins. This is the "Golden Ticket" attack. Rotating krbtgt requires changing the password twice (to invalidate all existing tickets) and is a major incident response step.

Kerberos Delegation — and Why It's Dangerous

Kerberos delegation allows a service to use your credentials to authenticate to another service on your behalf. The web server can access a database "as you."

Three types (from least to most dangerous):

1. UNCONSTRAINED delegation (avoid at all costs):
   Service can impersonate the user to ANY service.
   If the service is compromised → attacker can impersonate any user to any system.
   Still common on legacy DCs. Golden ticket equivalent if DC is delegated.

2. CONSTRAINED delegation (acceptable with care):
   Service can only impersonate users to specified target services.
   Configured by admins: "WEBAPP can only delegate to SQL-SERVER."

3. RESOURCE-BASED CONSTRAINED delegation (RBCD) — modern:
   Target service specifies who can delegate TO it (not the other way around).
   More flexible. But also abusable — if you can write msDS-AllowedToActOnBehalfOfOtherIdentity,
   you can configure delegation to your own attacker machine (AD privilege escalation path).

NTLM — Windows Challenge-Response (Legacy)

NTLM (NT LAN Manager) is the older Windows authentication protocol, predating Kerberos in Windows environments. It's still used as a fallback when Kerberos isn't available — workgroup machines, IP-address-based connections, some legacy apps. Unlike Kerberos, there's no trusted third party: authentication happens directly between client and server.

NTLM Challenge-Response Flow
💻
1. NEGOTIATE — Client announces capabilities
Client sends a NEGOTIATE message listing supported NTLM version and flags (NTLMv2 required in modern environments).
🎲
2. CHALLENGE — Server sends a random 8-byte nonce
server_challenge = CSPRNG(8 bytes)
This is what the client must prove it can respond to correctly.
📝
3. AUTHENTICATE — Client computes and sends response
NTLMv2 Response:
NT_hash = MD4(UTF-16LE(password))
NTLMv2_hash = HMAC-MD5(NT_hash, username + domain)
response = HMAC-MD5(NTLMv2_hash, server_challenge + client_challenge + timestamp)
Client also generates its own client_challenge (prevents pre-computation).
✅
4. Server (or DC) verifies the response
If domain-joined: server passes the challenge + response to the Domain Controller for verification via NetLogon. DC recomputes the expected response and compares.

Why NTLM Is Dangerous and Should Be Disabled

NTLM Relay Attack

Attacker sits between client and server. Client sends NTLM credentials to attacker's machine (e.g., via a malicious UNC path in a phishing email). Attacker relays those credentials to a real service (e.g., a domain controller or Exchange). Because NTLM is a direct client-server protocol with no mutual authentication guarantee, the attacker relays valid credentials without ever knowing the password. Tools: Responder, ntlmrelayx.

This attack requires no cracking — it's real-time relay.

Pass-the-Hash

NTLM uses the NT hash as the key — you don't need the actual password. An attacker who extracts the NT hash from LSASS memory (Mimikatz) can authenticate as that user to any NTLM service. The "hash" is functionally equivalent to the password itself.

MD4 is not a password hash — it has no salt, no cost factor. Always disablable in modern AD.

Microsoft's trajectory: Windows 11 and Server 2025 allow administrators to fully disable NTLM. Use Kerberos for domain auth. For environments that can't fully disable NTLM, enable EPA (Extended Protection for Authentication) and SMB signing to prevent relaying.

RADIUS & LDAP — Network and Directory Authentication

RADIUS — Remote Authentication Dial-In User Service

RADIUS (RFC 2865) is the protocol behind network access authentication — VPNs, Wi-Fi (802.1X), and network switches authenticating users. When you connect to corporate Wi-Fi and it asks for your domain credentials, RADIUS is the protocol talking to the backend identity store.

RADIUS Authentication Flow (802.1X Wi-Fi)
💻
1. Client tries to connect to Wi-Fi
Your laptop connects to the corporate SSID. The access point initially blocks all traffic except EAP (authentication) frames.
📡
2. Access Point forwards credentials to RADIUS server
The AP (acting as RADIUS client / authenticator) wraps the EAP exchange in RADIUS packets and sends them to the RADIUS server (e.g., Microsoft NPS, FreeRADIUS, Cisco ISE).
RADIUS traffic is protected with MD5(shared_secret + request_authenticator) — weak by modern standards; use over TLS (RADSEC) in new deployments.
🔒
3. RADIUS server validates via identity backend
RADIUS server checks credentials against Active Directory (via LDAP/Kerberos), a local user database, or a certificate store (for EAP-TLS). Returns Access-Accept or Access-Reject.
✅
4. Access Point opens the port
On Access-Accept, the AP allows network traffic. RADIUS also returns VLAN assignments and other policies.

LDAP — Lightweight Directory Access Protocol

LDAP is not really an authentication protocol — it's a directory access protocol for querying and modifying directory services (like Active Directory). But it's widely used for authentication via "LDAP bind" operations.

LDAP Bind — How Apps Authenticate Against AD
💻
1. App connects to LDAP server (port 389 or 636 for LDAPS)
Always use LDAPS (LDAP over TLS) or STARTTLS. Plain LDAP sends credentials in cleartext — never acceptable in production.
👤
2. Simple Bind — app sends username + password
BIND { DN: "cn=alice,dc=corp,dc=com", password: "s3cr3t" }
AD verifies the credentials. Returns success or failure. The app now knows Alice authenticated.
✅
3. App queries directory for attributes
Once bound, app queries: "What groups is Alice in? What's her email? Is she active?" — used for authorization decisions and profile data.
LDAP injection: If an application builds LDAP queries using user-supplied input without sanitization, an attacker can modify the query logic — similar to SQL injection but against a directory service. Always use parameterized LDAP queries and a well-tested LDAP library.

Smart Cards, PIV & CAC — Hardware Certificate Authentication

A smart card is a physical card containing a microchip that stores a private key and an X.509 certificate — and critically, the private key never leaves the chip. When you insert the card, the chip performs the cryptographic operation internally. Even the card reader never sees the private key.

Smart Card Authentication Flow
🔒
Setup: Card is provisioned with a key pair + certificate
An enterprise CA issues an X.509 certificate for the user. The private key is generated on the chip and never exported. Certificate contains the user's identity (name, employee ID, email).
💳
Login: Insert card + enter PIN
The PIN unlocks the card's private key for use. The PIN stays on the card — it's never sent to the computer or network. After N failed PIN attempts, the card locks permanently (tamper protection).
📝
Windows sends a challenge to the card
The OS sends a random challenge. The smart card chip signs it using the stored private key (ECDSA or RSA) — the key never leaves the chip. Signed response returned to OS.
🎫
Kerberos PKINIT — certificate becomes Kerberos ticket
Windows uses the card's signature (via PKINIT extension to Kerberos) to get a TGT from the DC. The DC verifies the certificate against the enterprise CA. From here, normal Kerberos tickets flow.
✅
Authenticated — two-factor by design
Factor 1: Something you have (the physical card). Factor 2: Something you know (the PIN). Both required. If card is stolen without PIN, useless. If PIN is known without card, useless.

The key property all hardware tokens share: The private key is generated on-device and physically cannot be extracted. Even if you hand the card/key to an attacker, they cannot get the private key out — they would need to physically destroy the chip (destroying the key in the process). This is what "hardware-bound" means.

API Keys

API keys are the simplest and most widely deployed machine authentication mechanism — a long random string that acts as both identity and secret. They're used by virtually every public API (GitHub, Stripe, AWS, OpenAI). They're simple to implement, simple to use, and have well-known security weaknesses.

API Key — What It Is Cryptographically
🎲
Generation: a CSPRNG string with a prefix
key = prefix + "_" + base62(CSPRNG(32 bytes))
Example: sk-live_4bK9mN2pQrX7wYvZ...
The prefix (sk-live) lets secret scanning tools identify leaked keys automatically. GitHub, Stripe, and AWS all use this pattern.
💾
Storage: server stores a hash, not the raw key
stored = SHA-256(api_key) (no slow hash needed — key is long and random, no dictionary attacks)
The raw key is shown to the user exactly once at creation. If lost, a new key must be generated.
📨
Usage: sent in every request header
Authorization: Bearer sk-live_4bK9mN...
Server hashes the received key → looks up the hash → grants access if found. No session state needed.

API Key Security Properties and Weaknesses

Properties

Simple to implement and understand
Stateless (hash lookup = O(1))
Easy to scope and rotate
Works for machine-to-machine without user interaction
No crypto libraries needed by caller

Weaknesses

Bearer token — possession = access, no proof of identity
Easily leaked (logs, git history, environment vars)
No expiry by default — compromised keys live forever
No audience binding — leaked key works anywhere
No forward secrecy — old requests can be replayed
Requires out-of-band distribution (how does Alice get the key securely?)

# Send: key_id + timestamp + signature
# Server re-derives signature — key never travels on the wire
# Provides: replay protection (timestamp), integrity (body hash)
# This is how AWS Signature Version 4 works.

Magic Links & Passwordless Email Auth

A magic link is a one-time URL sent to a user's email that logs them in automatically when clicked. The email inbox acts as the proof of identity — "if you can receive email at this address, you are who you claim to be." No password required.

Magic Link Flow
📧
1. User enters email address
No password field. Just an email address. The claim: "I am alice@example.com."
🎲
2. Server generates a signed token
token = CSPRNG(32 bytes) stored in DB with expiry (15 minutes).
Or a signed JWT: { email, exp: now+15min, nonce } signed with server's key.
link = https://app.com/auth?token=TOKEN emailed to alice@example.com.
✉️
3. Link proves email access
Only someone with access to alice@example.com's inbox can click the link. The email channel is the second factor.
✅
4. Server verifies token and creates session
Token must: match stored value (or verify JWT signature), not be expired, not have been used before (one-time use). Creates a regular session cookie. Token is deleted/invalidated immediately.

Device Trust & Continuous Authentication

Traditional authentication is a point-in-time event: you log in once and get a session. Device trust and continuous authentication extend this: the system continuously evaluates whether the device and user are still trustworthy, not just at login time.

Device Trust — Cryptographic Device Identity

How Device Trust Works
💻
1. Device gets a cryptographic identity certificate
When a device is enrolled (MDM/Intune/Jamf), the endpoint generates a key pair. The private key is stored in the TPM (Trusted Platform Module) — a hardware chip that cannot be extracted.
The org's CA issues a device certificate: CN=LAPTOP-4827, OU=Managed Devices
📋
2. Device proves identity on every connection
When connecting to corporate resources, the device presents its certificate (via mTLS or device posture check). The certificate proves: "This is a specific managed device, enrolled with the org."
📋
3. Device posture is evaluated
Beyond identity, the system checks: Is disk encryption enabled? Is the OS patched? Is EDR running? Is the screen locked? These "posture signals" are attested by the device agent and factored into the access decision.
✅
4. Access granted based on device + user + posture
Zero trust policy: user == alice AND device == managed AND device.encrypted == true AND device.os_version >= 14.0. All conditions must be met.

The TPM — Hardware Root of Trust

The TPM (Trusted Platform Module, ISO/IEC 11889) is a hardware security chip on virtually all modern PCs and servers. It provides:

TPM CapabilityHow It's Used
Hardware-bound keysPrivate keys generated and stored inside TPM. Cannot be exported. Device cert key in device trust is TPM-bound.
Platform measurements (PCR)TPM measures the boot chain (BIOS → bootloader → OS kernel) and stores hashes in Platform Configuration Registers. Remote attestation proves the device booted cleanly.
Disk encryption key sealingBitLocker seals the disk encryption key to specific PCR values. If the boot chain changes (e.g., bootkit), TPM refuses to release the key.
Remote attestationTPM can produce a signed quote of its PCR values. A remote verifier can check: "Did this device really boot with a clean OS?" Used in confidential computing.

Continuous Authentication

Rather than authenticating once at login, continuous authentication monitors behavioral signals throughout the session and re-challenges or terminates the session when risk increases.

Continuous Auth Risk Signals
Behavioral Biometrics

Typing cadence, mouse movement patterns, touch pressure on mobile. ML model trained on the user's behavior. Deviation triggers re-auth or session termination.

Risk Signals

New IP address or country, unusual access hours, privilege escalation, accessing sensitive resources for the first time. Scored in real time. High risk → step-up MFA.

Cryptographic Binding

Session tokens can be bound to device certificates (mTLS) or browser keys (Token Binding / DPoP). If the session token is stolen and used from a different device, the cryptographic binding fails.

The Zero Trust principle: "Never trust, always verify." Continuous authentication is the technical implementation of this: every request is evaluated independently — user identity, device trust, posture, network location, and behavior — not just at login but throughout the session. The session token itself is not sufficient trust; it's an input to a continuous policy evaluation.

    ├── Can't avoid passwords? → Argon2id + TOTP (FIDO2 preferred over TOTP)
    └── Need MFA second factor?
        ├── Best → FIDO2 hardware key (YubiKey)
        ├── Good → TOTP authenticator app
        └── Acceptable → Push notification with number matching (not SMS)
The unifying thread through all of these: Every strong authentication method eventually reduces to the same cryptographic primitive — a digital signature. The entity being authenticated proves it holds a private key by signing a fresh challenge. Whether it's Kerberos signing an authenticator, SSH signing a nonce, WebAuthn signing a challenge, or a smart card signing via PKINIT — it's all the same operation. The differences are in how keys are managed, how trust is established, and what the challenge is bound to.

Web Application Firewalls (WAFs)

A Web Application Firewall sits between the internet and your application, inspecting HTTP/S traffic and blocking requests that match known attack signatures. WAFs operate at Layer 7 (application layer) and complement — but do not replace — secure application code.

How WAFs Work

WAFs use two primary detection modes:

  • Signature-based (negative security model): Maintain a database of known attack patterns (SQLi keywords, XSS payloads, shell commands). Block requests matching these signatures. Fast but bypassable — an attacker who knows the signatures can craft payloads that evade them.
  • Anomaly-based / behavioural (positive security model): Learn what normal traffic looks like, then block deviations. Lower false-negative rate but higher false-positive rate during training. Used in enterprise WAFs like AWS WAF v2 with ML rules.

DDoS Mitigation

A Distributed Denial of Service attack overwhelms a target with traffic from many sources simultaneously. Modern attacks are multi-vector — combining volumetric, protocol, and application-layer floods.

Attack Categories

L3/L4 (Network)VolumetricUDP flood, ICMP flood, amplification (DNS, NTP, memcached)Upstream scrubbing, anycast routing, BCP38 filtering
L4 (Transport)ProtocolSYN flood, ACK flood, fragmentation attacksSYN cookies, connection rate limits, stateful firewall
L7 (Application)ApplicationHTTP GET/POST flood, Slowloris, RUDY, CC attacksRate limiting, challenge pages, behavioural analysis

SYN Flood & SYN Cookies

# Normal TCP 3-way handshake:
Client → SYN → Server (allocates half-open connection in table)
Client ← SYN-ACK ← Server
Client → ACK → Server (connection established)

# SYN flood: attacker sends SYN with spoofed source IPs — server allocates
# connection table entries that never complete → table fills → legitimate
# connections refused

# SYN cookies mitigation:
# Server does NOT allocate state on SYN.
# Instead: encodes connection parameters in the ISN (Initial Sequence Number)
# of the SYN-ACK using a cryptographic hash.
# When the ACK arrives with ISN+1, server verifies and recreates state.
# Result: no per-connection memory allocated until ACK confirms legitimacy.

L7 Application-Layer Mitigation

Slowloris: opens many connections, sends headers slowly — keeps connections open
  → Fix: set aggressive read timeouts; limit concurrent connections per IP

HTTP flood: sends legitimate-looking GET/POST at high rate
  → Fix: rate limiting per IP/user; CAPTCHA/JS challenge for suspicious IPs

CC (Challenge Collapsar): uses proxy networks, mimics real browsers
  → Fix: behavioural fingerprinting; TLS fingerprinting (JA3 hashes);
         device integrity signals; WAF bot management

Mitigation Architecture

  • Anycast scrubbing: Traffic routed to nearest PoP, malicious traffic filtered before reaching origin (Cloudflare Magic Transit, Akamai Prolexic)
  • BGP blackholing: Announce attacked IP to upstream providers for null-routing — drops all traffic including legitimate (emergency measure)
  • Rate limiting at edge: Cloudflare/AWS WAF rate limit rules before traffic hits origin servers
  • Origin IP protection: Keep origin IP secret; route all traffic through CDN/proxy; use Cloudflare Aegis or AWS Shield for origin protection

Network Segmentation & Zero Trust

Traditional flat network:
  Internet → Firewall → DMZ → Internal network (one perimeter)
  Once inside DMZ: attacker can laterally move to databases, internal APIs, AD

Segmented (defence-in-depth):
  Internet → Edge FW → DMZ (web servers)
                    → App tier (app servers) — only reachable from DMZ
                    → Data tier (DBs) — only reachable from app tier
                    → Management (bastion) — only reachable from corp VPN

Zero Trust principles:
  - Never trust, always verify — authenticate every request
  - Least privilege access — services can only reach what they need
  - Microsegmentation — each service has its own firewall rules
  - Assume breach — design so compromise of one service doesn't give full access
Cloud security groups: AWS Security Groups are stateful firewalls at the instance level. Set inbound rules to allow only required ports from required sources. Use VPC flow logs for network visibility.

CDN Security

Content Delivery Networks act as a reverse proxy between the internet and your origin servers, providing performance, availability, and security benefits. Understanding their security model is important for both exploiting and defending CDN-backed applications.

Security Benefits

  • TLS offloading: CDN terminates TLS at the edge — handles certificate management, OCSP stapling, TLS 1.3, HTTP/3
  • Origin IP protection: Real origin IP hidden behind CDN IP ranges — prevents direct-to-origin DDoS bypassing CDN protections
  • DDoS absorption: Distributed edge network absorbs volumetric attacks before they reach origin
  • WAF integration: Edge WAF rules applied globally at CDN layer
  • Rate limiting: Per-IP or per-endpoint rate limits enforced at edge

Cache Poisoning

Web cache poisoning: attacker causes the CDN to cache a malicious response
and serve it to subsequent legitimate users.

Classic example — unkeyed header injection:
  GET /api/data HTTP/1.1
  Host: example.com
  X-Forwarded-Host: evil.com   ← not part of cache key

  Response (cached):
  Location: https://evil.com/data   ← redirect to attacker site

  All subsequent requests to /api/data get the cached malicious redirect.

Fix:
  - Include all relevant headers in the cache key
  - Never reflect unvalidated headers into responses
  - Set Vary header correctly: Vary: Accept-Encoding (include headers used in response)
  - Use Cache-Control: no-store for sensitive or user-specific responses
